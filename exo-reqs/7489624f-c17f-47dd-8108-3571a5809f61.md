# req(exo): status-transition button never crashes on a non-Task/Project/Meeting effort class

## Job Story

When I click a generic status-transition button (Rollback to Backlog / Start / Done) on any asset that carries `ems__Effort_status` — including effort-like classes that are NOT `ems__Task`/`ems__Project`/`ems__Meeting` (e.g. `ems__Action`, `ems__Bug`, `ems__Idea`, `ems__OfflineMeeting`) — I want the command to either perform the transition or degrade gracefully, so I never see a "Command failed: … no recognised ems__* class" crash.

## Statement (Gherkin)

```gherkin
Scenario: graceful no-op on a class with no status workflow
  Given an asset whose exo__Instance_class is a UID-canon class outside Task/Project/Meeting
    And the class has no applicable status workflow (no ems__Effort_workflow override, no ems__Workflow ABox)
  When a workflow_transition grounding runs against it
  Then the result is a graceful no-op (notApplicable) — NOT a "no recognised ems__* class" failure

Scenario: data-driven transition for a non-built-in class that DOES have a workflow
  Given an asset of a non-built-in class with a declared workflow (e.g. via ems__Effort_workflow)
    And the asset is in Doing
  When the Rollback-to-Backlog workflow_transition runs
  Then its status is mutated to Backlog (the transition is resolved from vault data, not a hardcoded class map)
```

## Verification

**Revert-verified (unit):** `@req:7489624f-c17f-47dd-8108-3571a5809f61` — reverting `GroundingExecutor.executeWorkflowTransition` to the pre-fix hardcoded 3-class gate (return "no recognised ems__* class" for any class outside Task/Project/Meeting) makes both `GroundingExecutor.workflow_transition.test.ts` `@req:7489624f` cases go **RED** (graceful no-op expected `notApplicable=true` but got the fail-loud error; the non-built-in transition expected `success:true` but short-circuited); restored → **GREEN** (2026-06-29, wf-transition, origin/main `c98048a7`).

## Notes

- Resolution is data-driven (Homoiconicity): a per-asset `ems__Effort_workflow` override resolves a workflow for ANY class; Task/Project/Meeting keep their built-in class defaults (a vault `ems__Workflow` ABox with matching `Workflow_targetClass` overrides them when present).
- `ems__Action` (own Create/Execute/Trash one-shot lifecycle) resolves to `null` (no workflow) → graceful no-op, not given an Effort workflow.
- **Follow-up (out of scope here):** (a) hide generic status buttons via precondition gating on classes without a workflow (currently they render but no-op gracefully); (b) per-class `ems__Workflow` ABox resolution for arbitrary non-built-in classes; (c) add default workflows for `ems__Bug`/`ems__Idea`/`ems__OfflineMeeting` if they should be status-managed.
