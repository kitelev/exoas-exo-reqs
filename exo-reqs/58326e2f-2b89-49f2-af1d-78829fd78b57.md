# req(exo): active ui-acceptance is auto-verified by an eka-gui GUI-in-CI binding (automated sub-mode, dedup)

## Job Story

When a `ui-acceptance` requirement describes a GUI behavior that the `eka-gui-e2e`
runner can drive (a real plugin-UI scenario against a fresh vault on native-amd64
CI), I want to verify it by binding an eka-gui scenario to it with a `@req:<uid>`
tag — and have that CI run auto-produce + auto-attribute the evidence — so that an
automatable ui-acceptance requirement no longer needs a hand-written committed
evidence-attestation, ONE engine (eka-gui-e2e) serves both gui-bdd and automated
ui-acceptance, and no parallel ui-acceptance runner has to be built (dedup).

## Statement (Gherkin)

```gherkin
Given an active req__Requirement R with binding class ui-acceptance
When R is bound by a @req:<R-uid> tag in an eka-gui runner spec (tests/e2e/eka-gui/)
Then `exocortex requirements audit` treats R as satisfied via the AUTOMATED sub-mode
  (the eka-gui run is the auto-evidence — no committed req__Requirement_evidence needed)
And R is counted under uiAcceptanceAutomated (not uiAcceptanceManual)

Given an active ui-acceptance req R
When R has a resolving committed req__Requirement_evidence attestation and NO eka-gui @req binding
Then R is satisfied via the MANUAL sub-mode and counted under uiAcceptanceManual

Given an active ui-acceptance req R
When R is bound ONLY by a @req tag in a NON-GUI test (jsdom unit/component, CLI integration)
  and has no committed evidence
Then R is an active-requirement invariant violation
  (a non-GUI tag does not auto-evidence a live-UI acceptance)
```

## Verification

**Binding class: `unit`** — the satisfaction logic is a pure function
(`auditTraceability`) verified by a production-shape revert-verify executable
specification (`requirements-audit.test.ts`): the `TagOccurrence.file` path
distinguishes an `eka-gui` runner spec (automated) from a non-GUI test, and a
body-only revert of the GUI-runner-aware active-gate (restore the broad
`if (isBound) continue;`) makes the tightening test FAIL, restored → PASS. The
eka-gui auto-evidence production (manifest + screenshot) is corroborated by the
reporter unit test (`extractReqUids`).
