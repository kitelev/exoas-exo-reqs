# req(exo): CLI `apply` canonicalises the target path to vault-relative before precondition evaluation (absolute-path parity)

## Job Story

When I drive the gomoiconic task lifecycle from the CLI (e.g. executive-assistant automation that resolves task files to absolute paths), I want `apply start-effort`/`set-draft-status`/… to evaluate the command's SPARQL-ASK precondition against the asset I actually named — regardless of whether I spell the path absolute or vault-relative — so the transition succeeds when the asset genuinely satisfies it, instead of failing with a false-negative "Precondition not satisfied".

## Statement (Gherkin)

```gherkin
Given a vault with a status-transition command whose precondition gates on $target's current ems__Effort_status
And a Backlog ems__Task at <vault>/<assetspace>/<uid>.md
When I run `apply start-effort <ABSOLUTE-path-to-the-task>` against the vault
Then the precondition passes and the task's status is set to Doing (real mutation)

When I run `apply start-effort <vault-relative-path-to-the-task>` against the vault
Then the precondition passes and the task's status is set to Doing (same outcome — path-form parity)

Given a path that resolves outside the vault root
When I run `apply <cmd> <that-path>`
Then apply reports the target is outside the vault and performs no mutation (no crash, no phantom write)
```

## Verification

**Bound (integration):** `@req:a8d01c40-2902-45e2-ba28-501144ad1632` — `packages/cli/tests/integration/apply-absolute-path-precondition.integration.test.ts` runs the REAL `apply` pipeline (NoteToRDFConverter.convertVault → CommandResolver.loadCommand → PreconditionEvaluator.evaluate → GroundingExecutor) against a temp vault and reads the mutated file back. Revert-verified: reverting the `vaultPathToIRI(vaultRelative)` canonicalisation → ABSOLUTE case RED, RELATIVE case GREEN.

## Notes

- **Root cause (verify-before-assert, empirically confirmed):** the reporter's "ASK true in store but apply rejects" symptom was a `$target` subject-IRI mismatch produced by an absolute target path, NOT a precondition-logic bug. Reproduced deterministically: `apply set-draft-status <absolute>` → "Precondition not satisfied"; `apply set-draft-status <relative>` → passes. Both #3788 and #3789 are the same path-normalisation bug — once it is fixed, a no-status CLI task drives the full chain Draft→Backlog→Doing→Done.
- **#3789 decision (a/b/c):** generic `create` stays a low-level primitive (option ~c — no lifecycle default added to `create`, no public-contract change). The standalone-task lifecycle entry point already exists: `apply set-draft-status` (precondition = "has no status"). Resolved-by-#3788.
