# req(exo): `run-query <uid>` + homoiconic `classes` read-axis

> Source: homoiconization audit of exocortex-cli read-commands (orchestrator + Andrey interview 2026-07-11). Scope C ("new standalone `run-query <uid>` verb"). Axial framing (Vision Lock #1): in code stays only the engine/platform/FS; domain lives in vault on two axes — **write → `exocmd__Command`** (`apply`), **read → `query__NamedQuery`** (`NamedQueryRunner`). This is the **first CLI consumer of the read axis**.

## Job Story

When I have a read-only named query declared as vault data (a `query__NamedQuery` asset), I want to run it directly from the CLI (`run-query <uid>`) and see its result — instead of the query being reachable only implicitly from inside a precondition or grounding. And when the `classes` introspection command returns a class list or a class description, I want to change *what it returns* by editing a vault query asset, not by editing CLI code — so schema-introspection follows the same homoiconic "domain lives in vault" discipline as write commands.

## Statement (Gherkin)

```gherkin
Scenario: run-query executes a query__NamedQuery asset and prints its SELECT result
  Given a query__NamedQuery asset in the vault whose ```sparql body is a SELECT over the graph
  When I run `exocortex run-query <uid> --vault <vault>`
  Then the command resolves the asset by UID via FsQueryBodyResolver over the hydrated store
    And executes the SPARQL body read-only through NamedQueryRunner
    And prints the SELECT rows (an ASCII table by default; a JSON array with --format json)

Scenario: run-query prints an ASK result as a boolean
  Given a query__NamedQuery asset whose ```sparql body is an ASK
  When I run `exocortex run-query <uid> --vault <vault>`
  Then the command prints the boolean result (true/false)

Scenario: run-query fails closed on an unresolvable NamedQuery
  Given a UID that maps to no asset, or to an asset with no ```sparql block
  When I run `exocortex run-query <uid> --vault <vault>`
  Then the command exits with a clear error naming the UID (fail-closed), not a stack trace

Scenario: run-query substitutes a literal parameter into the query body
  Given a query__NamedQuery asset whose body contains a $className token
  When I run `exocortex run-query <uid> --param className=ems__Task --vault <vault>`
  Then $className is substituted as the escaped literal "ems__Task" before parse (engine-controlled substitution, read-only)

Scenario: classes reads its class-list query from a vault asset, not from hardcoded SPARQL
  Given a canonical query__NamedQuery asset that supplies the "list classes + instance counts" SPARQL
  When I run `exocortex classes --vault <vault>`
  Then the class list + counts printed are produced by executing THAT vault asset's SPARQL body
    And editing the asset's SPARQL body (e.g. adding a FILTER that excludes a class) changes what `classes` prints — with no code change
    And the ASCII table formatter (prefix__local truncation) stays in code; the output shape is preserved

Scenario: describe-class reads its count + properties queries from vault assets, parametrized by class name
  Given canonical query__NamedQuery assets for "instance count of a class" and "properties used by a class", each taking a $className literal parameter
  When I run `exocortex classes ems__Task --vault <vault>` (or the `describe-class` alias)
  Then the instance count and the property-usage table are produced by executing those vault assets' SPARQL bodies with $className bound to "ems__Task"
    And the formatter stays in code; the output shape is preserved
```

## Verification

**Binding class: Integration** — a production-shape CLI integration test (`packages/cli/tests/integration/run-query-namedquery.integration.test.ts`) tagged `@req:2678df55-ef74-4b45-a168-90db5e958882` builds a real temp vault (canonical query assets + a handful of class instances), hydrates it through the real `FileSystemVaultAdapter → NoteToRDFConverter → InMemoryTripleStore`, and drives the real `FsQueryBodyResolver → NamedQueryRunner` — no stubs.

Revert-verify axes (recorded in the PR body at implementation time):
- **run-query reads the body:** the fixture query asset's `sparql` body drives the printed result; changing the body (different SELECT / an ASK) changes the output → the bound assertion is body-specific and cannot be produced by any hardcoded query → **RED** if `run-query` ignored the vault body.
- **classes reads from vault, not hardcode:** the fixture class-list query carries a distinctive `FILTER` that excludes one class; `classes` output omits that class → asserting its absence is **GREEN** only when `classes` executes the vault asset. Reverting `classes` to the old hardcoded (unfiltered) SPARQL makes the excluded class reappear → **RED**.
