# req(exo): cli create sets default ems__Effort_status (Backlog) for status-bearing classes plus default createdBy and --status/--yes flags

## Job Story

When I create a fresh `ems__Task` / `ems__Project` (or any `ems__Effort` subclass) via `exocortex create`, I want it to land in the default `Backlog` status directly — the same status a Write-template gives for free — instead of having to run an undocumented two-command follow-up (`apply set-draft-status` → `apply move-to-backlog`). I also want the CLI to stamp the standard `exo__Asset_createdBy` (ExoAssistant, matching the help text) and to accept `--status` and `--yes` like every other subcommand, so the created asset carries sensible metadata with no extra ceremony.

## Statement (Gherkin)

```gherkin
Given a vault whose ems__Task class definition is a subclass of ems__Effort
When I run `cli create --class ems__Task --label X` with no --status flag
Then the created asset has ems__Effort_status set to the Backlog enum wikilink
And the created asset has exo__Asset_createdBy set to the ExoAssistant wikilink

Given the same vault
When I run `cli create --class ems__Task --label X --status Draft`
Then the created asset has ems__Effort_status set to the Draft enum wikilink

Given a vault whose concept__Concept class definition is NOT a subclass of ems__Effort
When I run `cli create --class concept__Concept --label X`
Then the created asset has no ems__Effort_status property
And the created asset still has exo__Asset_createdBy set to ExoAssistant

Given a non-status-bearing class
When I run `cli create --class concept__Concept --label X --status Draft`
Then the command fails with a clear error that --status only applies to status-bearing classes

Given I pass --created-by <my-uid>
When I run `cli create`
Then exo__Asset_createdBy is my-uid (the explicit value wins over the ExoAssistant default)

Given I pass --yes
When I run `cli create`
Then the command succeeds (—yes is accepted as a no-op, not 'unknown option')
```

## Scope note

Implemented entirely in `packages/cli` (`create.ts` + a new `EffortStatusResolver`): the bare `exocortex create` verb bypasses homoiconic groundings (per homoiconic-gap-triage), so set-time defaults on it are an engine concern (Homoiconicity Invariant Q3 — core-of-processing). The core `GenericAssetCreationService` is unchanged, so the plugin / `apply` / grounding paths stay byte-identical (they have their own PropertyDefault / grounding mechanisms). `exo__Asset_updatedAt` is already set on create (PR #3854/#3855) and is out of scope.
