# req(exo): the property editor's Relations section lists inline + reified relations, creates inline relations via a range-scoped picker, and deletes a relation in place

## Job Story

When I open the property editor on an asset, I want a Relations section that shows all of
that asset's outgoing relations — both the ordinary inline (frontmatter) ones and the
ones reified into a separate `exo__Statement` asset, with a marker telling me where each
reified statement lives — and lets me add a new relation (picking a predicate and a
range-scoped target) or remove an existing one right there at the point of the relation —
so that I can manage my asset's relations without hand-editing raw frontmatter or the
statement files, and without ever seeing the same relation listed twice.

## Statement (Gherkin)

```gherkin
Scenario: the Relations section lists outgoing inline and reified relations, deduplicated
  Given an asset A whose frontmatter has inline wikilink relations
    And the triple store has exo__Statement instances where A is the subject (reified relations)
  When the property editor's Relations section builds its list
  Then it shows A's inline relations (wikilink-valued frontmatter, excluding housekeeping keys)
    And it shows A's outgoing reified relations, each with the factual "reified · <AS>" marker
    And a relation present BOTH inline and reified appears ONCE, shown as inline (inline wins)
    And a reified relation whose object is a Literal is excluded (it is a property, not a relation)
    And an inline relation carries no marker

Scenario: create a new inline relation via a range-scoped picker
  Given the Relations section's create affordance
  When I pick a predicate and a target asset
  Then the picker's candidate targets are scoped to the predicate's range (not the whole vault)
    And confirming appends the relation as an inline wikilink to A's frontmatter
    And an existing multi-value predicate keeps its other values (the new value is appended, not replacing)

Scenario: delete a relation in place
  Given a relation row in the Relations section with a delete affordance
  When I delete an inline relation
  Then that single value is removed from A's frontmatter (sibling values of the same key are kept)
    And the key is dropped only when its last value is removed
  When I delete a reified relation
  Then its backing exo__Statement asset is deleted (verify-after-write)
    And all writes go through app.vault.modify / app.vault.delete (Desktop↔Mobile parity)

Scenario: scope boundary (Task 3.1 = list + create + delete)
  Then there is NO reify/de-reify toggle in this section (that is Task 3.2)
    And new relations are always created inline (reify-on-share is Task 3.2)
```
