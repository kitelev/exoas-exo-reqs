# req(testsys): a nightly Stryker mutation run below the break-budget fails the nightly mutation job

## Job Story

When the core/cli test suite drifts into "harness theater" — tests that execute
code but would not actually catch a bug introduced into it — I want the nightly
mutation job to turn red the moment a package's mutation score falls below its
break-budget, so that test-coverage gaps the line-coverage gate is blind to
become a visible, ratchetable signal instead of silent rot.

## Statement (Gherkin)

```gherkin
Given the nightly Stryker mutation workflow runs over packages/core and packages/cli
  And each package has a break-budget in scripts/mutation-budget.json
When a Stryker run produces a JSON report whose total mutation score
  (killed+timeout over killed+timeout+survived+no-coverage) for a package is
  BELOW that package's break-budget
  (e.g. an escaped mutant survives because a test was weakened or removed)
Then scripts/check-mutation-score.mjs exits 1 and the nightly mutation job is RED
  And the job-summary records the score + killed/timeout/survived/no-coverage counts
When the surviving mutants are killed (the suite is strengthened) so the score is
  AT or ABOVE the budget
Then the gate exits 0 and the nightly job is GREEN
And a missing or unparseable report exits 1 (fail-closed — no verdict is never green)
And this gate is NOT a required PR check — it runs only in the nightly cron
```

## Verification

**Revert-verified (integration binding):** `@req:7af848cd-b8a5-4842-b0fb-b01184a94b95`
in `packages/obsidian-plugin/tests/unit/build/check-mutation-score.test.ts`. The
test drives the REAL gate script (`scripts/check-mutation-score.mjs`) against
fixture Stryker JSON reports via `--report` + `MUTATION_BUDGET`: a report whose
score is below budget exits **1** (RED), a report at/above budget exits **0**
(GREEN), a missing report fails closed (exit 1), and a report with zero
score-relevant mutants fails closed. The binding-class is `integration` (the real
script run end-to-end over fixture reports, no mocks of the gate itself).

**End-to-end revert-verify (the nightly pipeline, run on-demand locally):** a real
Stryker run on `packages/core/src/utilities/DateFormatter.ts` scored **91.43%**
(96 killed / 7 survived) → gate at budget 88 = GREEN. Skipping the test's five
"mutation killing" `describe` blocks (an escaped-mutant scenario) dropped the
score to **76.19%** (7 → 23 survived) → gate at budget 88 = RED (exit 1);
restoring the blocks returned it to 91.43% = GREEN. This confirms the full
pipeline (Stryker → JSON report → score gate → red/green) end-to-end, on real
core source.
