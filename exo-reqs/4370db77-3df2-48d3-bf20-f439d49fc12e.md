# req(exo): Composite homoiconic preconditions — recursive boolean evaluation

> Source: onto-RFC `df602adc-7b1a-41a4-a238-f0d3a8b5738e` (v2, user-approved via interview + 4-agent review). Phase 4 (code, способ A — boolean composition).

## Job Story

When I author a command whose availability depends on several reusable atomic preconditions combined with boolean logic (AND / OR / NOT), I want the plugin to load that combinator tree from the vault and evaluate it recursively, so I can factor a shared ASK fragment into one reusable precondition and compose it — without rewriting SPARQL and without a broken sub-precondition silently showing a command it should hide.

## Statement (Gherkin)

```gherkin
Scenario: AND / OR / NOT combinators
  Given a command precondition that is an exocmd__AllPrecondition over child preconditions
  When the command availability is evaluated against a target asset
  Then the command is available iff EVERY child precondition is satisfied
  And an exocmd__AnyPrecondition is available iff at least one child is satisfied
  And an exocmd__NotPrecondition is available iff its single child is NOT satisfied
  And arbitrary nesting of these combinators is evaluated recursively

Scenario: fail-closed broken sub-precondition (Impl-HIGH regress-lock)
  Given a composite precondition one of whose children cannot be loaded
    (an unresolvable wikilink, a cyclic reference caught by the visited-set,
     an over-depth reference, or a malformed atomic with no evaluation source)
  When the command availability is evaluated
  Then that child is a broken sentinel evaluating to false
  And the command is HIDDEN (fail-closed) — never silently shown

Scenario: fail-open top-level boundary (backward compatible)
  Given a command whose top-level precondition reference does not resolve
    (or the command has no precondition at all)
  When the command availability is evaluated
  Then the command is AVAILABLE (fail-open) — unchanged from the pre-composite behaviour
  And this is the deliberate asymmetry: a broken CHILD is fail-closed, an
    unresolvable TOP-LEVEL is fail-open
```
