# req(exo): ExoSync push delivers any pending local delta — the mtime-manifest is a perf cache only and never causes an un-pushed edit to be skipped

## Job Story

When I edit an asset and then run `exosync pull` (which finds no remote change to that file) followed by `exosync push`, I want the edit to be delivered to the remote — not silently skipped while `push` reports `synced — pushed 0` — so that the zero-loss promise holds and I never have to `touch` a file to force its delivery.

## Statement (Gherkin)

```gherkin
Given an ASSET-mode AssetSpace synced with a warm mtime-manifest cache
  And I edit a file locally (content + mtime + size all change)
  And the remote has NO change to that file
When I run `pull` (which pulls nothing) and then `push`
Then the edited blob is delivered to the remote (the remote head blobSha == the local edited blobSha)
  And `push` reports the file as pushed (pushedCount == 1), never `synced` with the edit undelivered
  And a follow-up no-op sync sees no residual local delta (push and the watermark base agree)
  And a genuinely in-sync file is still a cache hit on a no-op sync (the perf optimisation is preserved)
```

## Verification

**Revert-verified (integration binding):** `@req:702e8bda-1e13-42b0-9858-3d5697f71b6a` — the production-shape integration test drives the real `SyncEngine` (`FakeGitHubRepo`, a stat-tracking local port mirroring a real filesystem, an in-memory manifest store mirroring `FileLocalManifestStore`'s contract) through the exact issue scenario: warm cache → edit before `pull` → `pull` → `push`, asserting the remote received the edited blob. Removing the `cached.blobSha === baseBlobByPath.get(path)` clause from the cache-hit gate in `SyncEngine.ts` (type-preserving) makes the two behavioural tests go **RED** (push delivers nothing, the remote stays stale); restoring it → **GREEN** (2026-06-27, issue #3751, origin/main `2f03ceda`). The third test (cache-hit fast path preserved for in-sync files) stays GREEN in both states — it guards against a perf regression, not the bug, so it is not revert-bound.

> Fix for GitHub issue [#3751](https://github.com/kitelev/exocortex/issues/3751) (ExoSync: `push` after `pull` silently skips a pending-local-edit). Root cause: the mtime-manifest is rebuilt+persisted in the `pull` cycle from the post-edit disk snapshot, advancing the cache baseline past an un-pushed edit; the next `push` got a `(mtime,size)` cache hit, reused the cached blobSha WITHOUT reading the file, detection correctly flagged it modified-vs-base, but `pinLocalChanges` builds the push set from the read `disk` (which a cache-hit file is absent from) → the change was silently dropped (`ParityValidator`, which hashes independently, correctly saw the `pending-local-edit` — the asymmetry was the smoking gun). The fix gates the cache-hit skip on `cached.blobSha === watermark base`, restoring the documented invariant "a stale/divergent entry causes a re-hash (cache miss), NEVER a wrong skip" — correctness of delivery no longer depends on the perf cache. Binding-class `integration` (real engine over a fake remote, the production sync path under test) satisfies the P0 binding-class floor (RFC 0003 §3.6).
