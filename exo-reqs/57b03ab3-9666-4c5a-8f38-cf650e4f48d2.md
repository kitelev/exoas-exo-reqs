# req(exo): create_instance auto-inherits its prototype's declared time-of-day (ec15f83e)

## Job Story

When I tap "Create Task Instance" (or run `apply create-task-instance`) on a recurring meal/routine prototype that already declares its time-of-day on the prototype itself (`ems__EffortPrototype_startTime: "09:45"`, `ems__EffortPrototype_endTime: "10:15"`), I want the new instance to be planned for that time automatically, so that I declare the time ONCE on the prototype and every generated instance lands in the right slot of the day — instead of hand-setting `ems__Effort_plannedStartTimestamp` on each instance after creating it.

## Statement (Gherkin)

```gherkin
Scenario: prototype declares both start and end time → instance gets both planned timestamps
  Given an ems__TaskPrototype that declares ems__EffortPrototype_startTime "09:45" and ems__EffortPrototype_endTime "10:15"
  When a create_instance grounding materialises an instance from that prototype for a given day D
  Then the created instance frontmatter contains ems__Effort_plannedStartTimestamp = "D T09:45:00" (local, no timezone offset)
  And the created instance frontmatter contains ems__Effort_plannedEndTimestamp = "D T10:15:00" (local, no timezone offset)

Scenario: prototype declares only a start time → instance gets only plannedStart
  Given an ems__TaskPrototype that declares ems__EffortPrototype_startTime "20:00" and no ems__EffortPrototype_endTime
  When a create_instance grounding materialises an instance from that prototype for a given day D
  Then the created instance frontmatter contains ems__Effort_plannedStartTimestamp = "D T20:00:00"
  And the created instance frontmatter contains no ems__Effort_plannedEndTimestamp

Scenario: prototype declares no time-of-day → instance is unchanged (no regression)
  Given an ems__TaskPrototype that declares neither ems__EffortPrototype_startTime nor _endTime
  When a create_instance grounding materialises an instance from that prototype
  Then the created instance frontmatter contains no ems__Effort_plannedStartTimestamp written by this feature
  And the instance is identical to current behavior

Scenario: the stamped timestamp uses the instance's own date and is timezone-naive
  Given an ems__TaskPrototype declaring ems__EffortPrototype_startTime "14:15"
  When a create_instance grounding stamps the instance for date D (the same date used for the instance label / $today)
  Then ems__Effort_plannedStartTimestamp equals "DT14:15:00" with NO trailing timezone offset (e.g. not "+05:00", not "Z")
  And the value is a FULL date+time timestamp ("YYYY-MM-DDTHH:MM:SS") — NEVER the bare "HH:MM" time copied from the prototype (ems__Effort_plannedStartTimestamp is a dateTime, the prototype's ems__EffortPrototype_startTime is a time-of-day; the feature combines the instance date with that time-of-day to form the full dateTime)
```

## Verification (plan)

**Binding class:** `Integration` — the behavior is `GroundingExecutor.executeCreateInstance` reading the resolved prototype's frontmatter (`ems__EffortPrototype_startTime`/`_endTime`) and writing the instance frontmatter, exercised against a real prototype fixture + real FrontmatterService, not a stub. Test titles carry `@req:57b03ab3-9666-4c5a-8f38-cf650e4f48d2`. Both the plugin path and the CLI `apply create-task-instance` path are covered (parity).

**Revert-verify** ([[integration-test-revert-verify]]): with the propagation code neutralised in a type-preserving way (skip the prototype-time read / don't write the planned timestamps), the "prototype declares start+end → instance gets both" assertion goes **RED** (no plannedStart on the instance); restored → **GREEN**. The "prototype declares no time → unchanged" scenario stays **GREEN both ways** (guards against over-eager stamping — a test that passes both ways there is the intended no-op signal, not a false positive).

> Realises vault backlog idea `ec15f83e` (12 votes). Design note (Homoiconicity Invariant): this is core processing of the generic prototype-instantiation path — "an instance inherits its prototype's declared time-of-day" is a domain-agnostic rule over the generic `ems__EffortPrototype_startTime` property, so it lives in the `create_instance` executor (code), not per-grounding RDF. Out of scope: a recurrence ENGINE that auto-generates a whole week of instances (the existing `d86988cb` automation prompt + manual batch remain the weekly-fill mechanism); this requirement only makes a SINGLE create_instance inherit the prototype's time.
