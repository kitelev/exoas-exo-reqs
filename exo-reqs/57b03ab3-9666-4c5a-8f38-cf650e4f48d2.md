# req(exo): time-bearing prototype instance — choose target date (default today) + auto-stamp full planned timestamp (ec15f83e)

## Job Story

When I tap "Create Task Instance" (or run `apply create-task-instance`) on a recurring meal/routine prototype that declares its time-of-day on the prototype itself (`ems__EffortPrototype_startTime: "09:45"`, `ems__EffortPrototype_endTime: "10:15"`), I want to (a) pick which DAY the instance is for — defaulting to today, so the fast path stays "today" — and (b) have the instance automatically planned at the prototype's time on that day, so that I declare the time ONCE on the prototype and create a correctly-planned instance for ANY day (today, or any day next week) in a single action — instead of hand-setting `ems__Effort_plannedStartTimestamp` on each instance, and instead of only ever being able to create for "today".

## Statement (Gherkin)

```gherkin
Scenario: default target date is today
  Given an ems__TaskPrototype declaring ems__EffortPrototype_startTime "09:45" and ems__EffortPrototype_endTime "10:15"
  When a create_instance grounding materialises an instance from it WITHOUT an explicit target date
  Then the target date defaults to today (D = today)
  And the created instance frontmatter contains ems__Effort_plannedStartTimestamp = "DT09:45:00"
  And the created instance frontmatter contains ems__Effort_plannedEndTimestamp = "DT10:15:00"

Scenario: explicit target date is honoured (plugin modal date field / CLI date param)
  Given an ems__TaskPrototype declaring ems__EffortPrototype_startTime "09:45" and ems__EffortPrototype_endTime "10:15"
  When the user chooses target date D = "2026-07-01" (via the create modal's date field, or the CLI date parameter)
  Then the instance is stamped for D: ems__Effort_plannedStartTimestamp = "2026-07-01T09:45:00"
  And ems__Effort_plannedEndTimestamp = "2026-07-01T10:15:00"
  And the instance label / date denote D (not today)

Scenario: prototype declares only a start time → instance gets only plannedStart
  Given an ems__TaskPrototype declaring ems__EffortPrototype_startTime "20:00" and no ems__EffortPrototype_endTime
  When a create_instance grounding materialises an instance for date D
  Then the created instance contains ems__Effort_plannedStartTimestamp = "DT20:00:00"
  And the created instance contains no ems__Effort_plannedEndTimestamp

Scenario: prototype declares no time-of-day → unchanged (no regression, no date prompt)
  Given an ems__TaskPrototype declaring neither ems__EffortPrototype_startTime nor _endTime
  When a create_instance grounding materialises an instance from it
  Then no target-date prompt is shown and no date parameter is required
  And the created instance contains no ems__Effort_plannedStartTimestamp written by this feature
  And the instance is identical to current behavior

Scenario: the planned value is a FULL date+time, timezone-naive
  Given an ems__TaskPrototype declaring ems__EffortPrototype_startTime "14:15" and target date D
  When the instance is stamped
  Then ems__Effort_plannedStartTimestamp = "DT14:15:00" — a FULL date+time ("YYYY-MM-DDTHH:MM:SS"), NEVER the bare "HH:MM" from the prototype
  And it carries NO trailing timezone offset (not "+05:00", not "Z")
  And note the type distinction: ems__Effort_plannedStartTimestamp is a dateTime, ems__EffortPrototype_startTime is a time-of-day; the feature combines the chosen date with that time-of-day
```

## Verification (plan)

**Binding class:** `Integration` — core behavior is `GroundingExecutor.executeCreateInstance` reading the resolved prototype's `ems__EffortPrototype_startTime`/`_endTime` and writing the instance frontmatter with the chosen-date-or-today, exercised against a real prototype fixture + real FrontmatterService (not a stub). Test titles carry `@req:57b03ab3-9666-4c5a-8f38-cf650e4f48d2`. The CLI date-parameter path and the plugin modal date-field path are both covered (parity); the executor's date+time-combination + default-today logic is the integration-level heart.

**Revert-verify** ([[integration-test-revert-verify]]): with the propagation code neutralised in a type-preserving way (skip the prototype-time read / don't write the planned timestamps), the "default date / explicit date → instance gets plannedStart" assertions go **RED**; restored → **GREEN**. The "prototype declares no time → unchanged" scenario stays **GREEN both ways** (intended no-op signal, not a false positive — guards against over-eager stamping / spurious date prompts).
