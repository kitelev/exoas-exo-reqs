# req(exo): the `pn__DailyNote` tasks table renders task labels via the homoiconic `exo__DisplayNameSpec` system

> Source: this session's investigation (2026-07-11) — «почему в layout `pn__DailyNote` не отображается 🔄 у задач в статусе Doing». Root cause: the homoiconic displayName mechanism (`exo__DisplayNameSpec` + `DisplayNameResolver` + the per-render matcher of req `[[ed4201d1-f9da-4142-b12a-5769d4c67f38]]`) is wired ONLY into the 8 patches over Obsidian's native link rendering. The DailyNote tasks table bypasses it: `DailyTasksRenderer` -> `helpers.ts:getDisplayName` re-implements the emoji in TS, and its `effortStatusStr === EffortStatus.DOING` (a naive strip of only the outer `[[ ]]`) never matches a UID-canon status value -> 🔄 is silently lost, while the Status column (via `getStatusLabel`, which normalizes all forms) still shows "Doing" correctly. This req makes the DailyNote tasks table homoiconic — single source of truth in vault `exo__DisplayNameSpec` data (Homoiconicity Invariant Q1). The 🔄-Doing spec `[[d069c316-fcd3-4c25-8f24-3109382b72cf]]` already exists in-vault; the Done/Trashed/Meeting specs are authored as part of this req so those prefixes are homoiconic too.
>
> **Scope (split, DailyTasks first):** THIS req covers only the DailyNote tasks table. The layout relation/query tables and the computed 🚩 blocked marker are explicit follow-ups (see `covers`).

## Job Story

When I open a DailyNote and look at its tasks table, I want each task's status prefix (🔄 while Doing, ✅ Done, …) to be **the same as everywhere else Exocortex renders that task and to be driven by my vault `exo__DisplayNameSpec` data** — so authoring one spec changes the label on every surface at once, and the DailyNote table never silently drops a prefix that native links show.

## Verification

**Revert-verified (component, production-shape):** the bound suite (`@req:a577316f-f2ff-469b-a6d3-ff946eccc68f`) drives the real DailyNote-table label path (a `DailyTasksRenderer` label computed over an in-memory vault carrying the real `exo__DisplayNameSpec` 🔄-Doing spec + its parts, resolved via the real `DisplayNameResolver` + `PrintNameRuleService.scanVault`) — no hand-injected label, no stubbed resolver.

Revert-verify axes (to record in the PR body at implementation time):
- Revert the DailyTasks renderer wiring (label falls back to the old hardcoded path) -> a UID-canon Doing task's cell loses the resolver-driven "🔄 " -> the bound assertion goes **RED**; restore -> **GREEN** ("🔄 <label>").
- Break the dual-IRI status normalization of the retained sort flag -> a UID-canon Doing task is no longer detected as Doing (sort/detection assertion) -> **RED**.

## Follow-ups (out of scope of this req)

- **layout relation/query tables** (`RelationsRenderer` / `UniversalLayoutRenderer` / `TableLayoutRenderer`) render task-link labels via plain `exo__Asset_label` with no prefix — a second requirement will route them through `DisplayNameResolver`.
- **🚩 blocked marker** — `BlockerHelpers.isEffortBlocked` is a computed predicate (reads `ems__Effort_blocker`, resolves that asset, checks ITS status); the current single-value `exo__DisplayNameSpec` matcher cannot express it. Separate engine-extension issue (computed / host-function matcher, or a future `exo__DisplayNameMatcher`).
