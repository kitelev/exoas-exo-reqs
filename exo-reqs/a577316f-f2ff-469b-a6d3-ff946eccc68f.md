# req(exo): custom renderers render task-link displayName via the homoiconic `exo__DisplayNameSpec` system

> Source: this session's investigation (2026-07-11) — «почему в layout `pn__DailyNote` не отображается 🔄 у задач в статусе Doing». Root cause: the homoiconic displayName mechanism (`exo__DisplayNameSpec` + `DisplayNameResolver` + the per-render matcher of req `[[ed4201d1-f9da-4142-b12a-5769d4c67f38]]`) is wired ONLY into the 8 patches over Obsidian's native link rendering. The plugin's OWN renderers bypass it: the `pn__DailyNote` tasks table re-implements the emoji in TS (`DailyTasksRenderer` → `helpers.ts:getDisplayName`, with a naive `effortStatusStr === EffortStatus.DOING` that a UID-canon status value never satisfies → 🔄 silently lost), and the layout relation/query tables (`RelationsRenderer` / `UniversalLayoutRenderer` / `TableLayoutRenderer`) render a plain `exo__Asset_label` with no prefix at all. This req makes those custom surfaces homoiconic — single source of truth in vault `exo__DisplayNameSpec` data (Homoiconicity Invariant Q1). The 🔄-Doing spec `[[d069c316-fcd3-4c25-8f24-3109382b72cf]]` already exists in-vault; the Done/Trashed/Meeting specs are authored as part of this req so those prefixes are homoiconic too.

## Job Story

When I look at a task link anywhere Exocortex renders it — the DailyNote tasks table, a relations table on any asset's layout, or a plain `[[task]]` in body text — I want its status/class prefix (🔄 while Doing, ✅ Done, …) to be **the same everywhere and driven by my vault `exo__DisplayNameSpec` data**, so that authoring one spec changes the label on every surface at once, with no per-renderer hardcode and no surface where the prefix silently goes missing.

## Verification

**Revert-verified (component, production-shape):** the bound suite (`@req:a577316f-f2ff-469b-a6d3-ff946eccc68f`) drives the real renderer path (a `DailyTasksRenderer` / relation-renderer over an in-memory vault carrying the real `exo__DisplayNameSpec` 🔄-Doing spec + its parts, resolved via the real `DisplayNameResolver` + `PrintNameRuleService.scanVault`) — no hand-injected label, no stubbed resolver.

Revert-verify axes (to record in the PR body at implementation time):
- Revert the renderer wiring (renderer falls back to the old hardcoded/plain-label path) → a Doing task's cell loses the resolver-driven "🔄 " (or renders the old broken/plain form) → the bound assertion goes **RED**; restore → **GREEN** ("🔄 <label>").
- Break the dual-IRI status normalization of the retained sort flag → a UID-canon Doing task is no longer detected as Doing (sort assertion / detection assertion) → **RED**.

## Open design point (for approval)

- **🚩 blocked marker.** Removing the renderer's hardcoded emoji map would also drop the computed 🚩. This req **retains 🚩 as a documented residual** (composed alongside the resolver output) so there is no regression, and defers the homoiconic expression of "blocked" to a separate engine-extension issue (computed / host-function matcher, or a future `exo__DisplayNameMatcher`). Confirm this residual approach vs. a stricter «remove all hardcode now (accept 🚩 regression until the engine work)».
