# req(exo): ExoSync change-detection tracks a moved asset by its stable uid (one modification, never a data-losing delete+add)

## Job Story

When I move or rename an asset's file on one device and then sync, I want ExoSync to recognise it as the SAME asset (by its stable `exo__Asset_uid`) and record a single modification (with its previous path), rather than seeing a deletion of the old path plus an addition of the new one — so a rename never destroys the asset's identity, history, or backlinks.

## Statement (Gherkin)

```gherkin
Given a sync base that recorded asset uid "u1" at path "old/a.md"
  And the working tree now holds the SAME uid "u1" at a new path "new/b.md"
When ExoSync's change detector diffs the working tree against the base
Then it derives exactly one MODIFIED change for "new/b.md" carrying basePath "old/a.md"
And it derives NO "added" and NO "deleted" change (no delete+add that would lose identity)
```

## Verification

**Revert-verified (unit binding):** `@req:8a97d6ed-9ffc-4802-8df5-09d1b371cb09` — disabling the uid-keyed Pass-1 lookup in `ChangeDetector.detectChanges` (`baseByUid.get(d.uid)` → a deliberately mismatched key, so the disk file never matches the base entry by uid) makes `ChangeDetector.test.ts "rename: same uid at a new path → modified with basePath, NOT delete+add"` go **RED** — the rename degrades to `added=[new/b.md]` + `deleted=[old/a.md]` (`modified` empty), exactly the data-losing delete+add the requirement forbids; restored → **GREEN** (2026-06-21, al-reqmgmt-p2-inc3, origin/main `53519a89`).

**Integration binding (floor, cited):** the full SyncEngine cycle runs `detectChanges` as its first step on every sync; the end-to-end change-detection path against real GitHub is covered by `syncEngine.real-github.integration.test.ts "pull→push: applies a disjoint device-B change and pushes the local one"` — a real `integration`-class test. That suite is **network/PAT-gated** (self-skips without `EXOSYNC_TEST_TOKEN` / `EXOSYNC_IT=1`, NOT in any CI allowlist by design), so a *dedicated* integration-class revert-verify is **deferred** to a manual `EXOSYNC_IT=1` run. At **P1** the revert-verified **unit** binding is the floor (P1 does not require a CI-reliable real-prod tier — RFC 0003 §3.6/§3.7); the declared `integration` class remains as a deferred end-to-end check, not the gating floor.
