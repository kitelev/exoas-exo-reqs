# req(exo): the property editor reifies/de-reifies a relation atomically, with predicate-mapping, multi-value, and object-IRI validation

## Job Story

When I look at a relation in the property editor's Relations section, I want to flip it between
an ordinary inline frontmatter relation and a relation reified into a separate `exo__Statement`
asset — so I can move a sensitive relation out of the asset (where it would travel on share)
into a statement asset I keep in a private AssetSpace, and pull it back when I want it inline
again — and I want that move to be atomic: I should never end up with the relation duplicated
(the inline copy left behind alongside the statement — a privacy leak) or lost (the inline copy
removed without the statement ever being written — data loss).

## Statement (Gherkin)

```gherkin
Scenario: reify an inline relation into a statement, atomically
  Given an inline relation A --predicateKey--> target in the Relations section
    And a predicate-definition asset exists whose label is predicateKey
  When I reify the relation
  Then an exo__Statement asset is created with _subject=A, _predicate=the predicate-def asset, _object=target
    And the statement is verified on disk before anything else changes
    And the single inline value is removed from A's frontmatter (sibling values of the key are kept)
    And the inline removal is verified before the relation is shown as reified
    And the relation now appears once, reified, with the "reified · <AS>" marker

Scenario: reify rolls back if the inline copy cannot be removed (no leak)
  Given the statement-create succeeds but removing the inline copy fails (or the inline copy is still present afterwards)
  When I reify the relation
  Then the just-created statement is deleted (rollback)
    And the inline copy remains intact (the edge is preserved, no duplicate, no leak)
    And the reify is reported as failed

Scenario: de-reify a reified relation back to inline, restoring before deleting
  Given a reified relation backed by an exo__Statement asset
  When I de-reify the relation
  Then the relation is first appended back to A's frontmatter as an inline wikilink (multi-value safe)
    And the inline copy is verified present BEFORE the statement is deleted
    And only then is the backing statement asset deleted and verified gone
    And if the statement delete fails the inline copy is still present (a transient duplicate, never a loss)

Scenario: predicate-mapping maps the frontmatter key to the predicate-definition asset
  Given an inline relation whose key has NO predicate-definition asset
  When I reify the relation
  Then the reify is rejected with an explicit error and no statement is created
  Given a reified relation
  When I de-reify it
  Then the predicate-definition asset's label is resolved back to the frontmatter key

Scenario: multi-value keys are handled one element at a time
  Given a multi-value relation key with several inline targets
  When I reify one of them
  Then only that one element is removed from the array (the others stay inline)
    And exactly one statement is created
  When I de-reify it
  Then exactly one element is appended back to the array

Scenario: a Literal object cannot be reified
  Given a relation whose object is a Literal (a scalar property value, not an IRI node)
  When I attempt to reify it
  Then the reify is rejected (a Literal is a property, not an edge) and no statement is created

Scenario: round-trip preserves edge identity
  When I reify an inline relation and then de-reify it
  Then the restored inline relation has the same uid-form target as the original
    And the materialized SPARQL property-path is unchanged

Scenario: scope boundary (Task 3.2 = toggle + atomicity)
  Then there is NO de-reify strong-confirm modal, NO object-side read-only, and NO destination-AS picker here (those are Task 3.3)
```
