# req(exo): CommandBinding targetPrototype matches the transitive prototype chain

## Job Story

When I scope a command button to a prototype family (e.g. «Mark Done Post Factum» on the «Покурить» prototype), I want one `exocmd__CommandBinding` with `targetPrototype = [[parent]]` to cover instances of the parent AND of every (transitive) child prototype, so I don't have to create and maintain a duplicate binding for each child prototype as the family grows.

## Statement (Gherkin)

```gherkin
Given a vault where prototype C has exo__Asset_prototype = [[P]] (C is a child prototype of P)
And an instance I_direct with exo__Asset_prototype = [[P]]
And an instance I_child with exo__Asset_prototype = [[C]]
And an exocmd__Command bound via an exocmd__CommandBinding with exocmd__CommandBinding_targetPrototype = [[P]]

When commands are resolved for I_direct (store built by NoteToRDFConverter.convertVault)
Then the bound command is present (direct-hop match — existing behavior preserved)

When commands are resolved for I_child
Then the bound command is present (chain-walk: I_child → C → P contains P)

When commands are resolved for an instance whose prototype chain does NOT contain P (or that has no exo__Asset_prototype)
Then the bound command is NOT present (no over-broadening)

Given a malformed vault where the prototype chain contains a cycle (A → B → A)
When commands are resolved for an instance of A
Then resolution terminates without hanging (visited-set cycle guard; depth cap) and matching considers only the visited chain
```

## Notes

- First live use of the `targetPrototype` mechanism at all: the resolver ships it since #3272-era work (constants + unit tests exist), but the vault contains 0 such bindings today; the exocmd TBox also lacks the `exocmd__CommandBinding_targetPrototype` property declaration — the TBox declaration + the «Mark Done Post Factum» command/binding data ship as a SEPARATE data-phase after this feature is merged and released.
- Dual-IRI caveat (sparql-iri-form-pre-verify): `exo__Asset_prototype` objects in the store are file-IRIs (`obsidian://vault/.../<uid>.md`), while the render path passes the direct prototype ref as a bare UID/wikilink — each chain hop must resolve the prototype reference to its store subject before reading the next `exo__Asset_prototype` triple.
- The walk lives in `CommandResolver` (store-side), NOT in the plugin's `extractPrototypeIRI` — all resolve surfaces (inline buttons via `DynamicCommandButtonGroupBuilder.resolveForAssetMulti`, `CommandRegistry.resolveForAsset`) inherit the semantics with no per-surface duplication.
- Resolver caching keys on `prototypeIRI` (`resolveForAssetMulti` cache key); the chain is derived deterministically from it + the store, so the existing cache-key shape stays valid; cache invalidation on store refresh is unchanged.
