# req(exo): `exo__DisplayNameSpec` gains a computed / host-function matcher (so 🚩 blocked can be homoiconic)

> Source: issue #3865, follow-up of req `[[a577316f-f2ff-469b-a6d3-ff946eccc68f]]`. The DailyNote table's 🚩 blocked marker is RETAINED renderer-side (`BlockerHelpers.isEffortBlocked`) because it is a **computed predicate** — it reads `ems__Effort_blocker`, resolves that asset, and checks ITS status. The current single-value `exo__DisplayNameSpec` matcher (matchPath/matchValue equality on the instance's own frontmatter) cannot express it. This req adds a computed / host-function matcher kind so cross-asset predicates become homoiconic `exo__DisplayNameSpec` conditions.
>
> **Engine-capability req — the most complex of the displayName line.** Reuses the existing `isEffortBlocked` host function; extends the matcher engine + the `exo__DisplayNameSpec` ontology with a host-function-matcher property. Delivers the CAPABILITY + a proof spec; the full renderer-residual removal is composition-gated (see the scope note in `covers`).

## Job Story

When I want a class's displayName to specialize on a CROSS-ASSET condition (e.g. a 🚩 prefix while an `ems__Task` is blocked — which depends on the referenced blocker's status, not the task's own frontmatter), I want to declare that condition as vault data on an `exo__DisplayNameSpec` (a host-function matcher) — so the prefix appears/disappears automatically as the computed condition changes, without renderer-side hardcode.

## Statement (Gherkin)

```gherkin
Background:
  Given a registered display-matcher host function "isEffortBlocked"
    (reuses BlockerHelpers.isEffortBlocked(app, metadata): true iff ems__Effort_blocker resolves to a not-done asset)

Scenario: a host-function-matched spec participates only when the computed predicate is true
  Given an exo__DisplayNameSpec for ems__Task with a PrintedLiteral "🚩 " + PrintedProperty label
    And it carries a host-function matcher naming "isEffortBlocked"
  When the plugin resolves the displayName of an ems__Task that IS blocked (its blocker is not done)
  Then the spec participates → "🚩 <label>" (subject to priority among participating specs)
  When the task is NOT blocked (no blocker, or the blocker is done)
  Then the spec does NOT participate → the label has no 🚩 from this spec

Scenario: the condition is evaluated per-render from the instance + app context (cross-asset)
  Given the 🚩-blocked spec is loaded once via scanVault
  When the referenced blocker asset's status changes from not-done to done
  Then a re-resolve no longer participates (the 🚩 disappears) — no re-scan required

Scenario: no matcher → v1 unconditional path byte-identical (no regression)
  Given a spec with NO matcher (neither matchPath/matchValue nor matchHostFunction)
  When the plugin resolves any asset of that class
  Then selection is identical to today (highest-priority unconditional spec wins)
```

## Verification

**Revert-verified (component, production-shape):** the bound suite (`@req:d6cd2371-bdf2-460e-840e-841480273869`) drives the real `PrintNameRuleService.scanVault → getTemplateForClass(metadata) → DisplayNameResolver.resolve` over an in-memory vault carrying a 🚩-blocked host-function spec + a task whose `ems__Effort_blocker` resolves to a not-done asset — no hand-injected matcher result; the real `isEffortBlocked` host function runs.

Revert-verify axes (record in the PR body at implementation time):
- Neutralize the host-function matcher evaluation (treat host-function specs as never-participating, OR always-participating) → the blocked task loses "🚩 " OR the non-blocked task gains a spurious "🚩 " → the bound assertion goes **RED**; restore → **GREEN**.
- A no-matcher spec assertion stays GREEN both ways (guards the v1 no-regression path).

## Out of scope (tracked)

- **Prefix composition** (a blocked Doing task showing BOTH 🚩 AND 🔄) — the single-winning-template model shows ONE; composition is the deferred #3864 follow-up. Therefore this req does NOT remove the renderer-side 🚩 residual (that awaits composition). File/keep a follow-up for the residual removal once composition exists.
- **#3864** multi-class membership — already shipped (v16.181.0).

## Design latitude

The exact ontology shape (a new `exo__DisplayNameSpec_matchHostFunction` property vs. a future `exo__DisplayNameMatcher` class) + the host-function registry shape are the implementer's design call, modeled on the existing matchPath/matchValue properties + the precondition host-function pattern. If the ontology addition feels like it needs its own onto-RFC, ESCALATE (WAITING_DECISION) rather than over-building; a minimal additive property reusing `isEffortBlocked` is the target.
