# req(exo): displayName prefix composition (completes homoiconic 🚩; removes the renderer residual)

> Source: issue #3878, the composition bundle deferred from #3865 (req `[[d6cd2371-bdf2-460e-840e-841480273869]]` / PR #3877 / v16.182.0, which shipped the host-function matcher CAPABILITY). The current model selects a SINGLE winning template, so a blocked Doing task can show 🚩 OR 🔄 but not both — which is why the DailyNote renderer still hardcodes a 🚩 residual (`daily-tasks/helpers.ts getDisplayName`) and why a live 🚩 spec was NOT shipped (it would double-🚩 blocked non-Doing tasks). This req adds prefix composition so 🚩 becomes fully homoiconic and the residual is removed.
>
> **The final + most-fundamental piece of the displayName line.** Changes the core selection model (single-winning-template → composition). If the composition model / its ontology feels RFC-scale, ESCALATE (see Design latitude).

## Job Story

When a task is BOTH blocked and Doing, I want its displayName to show BOTH markers (🚩 🔄) everywhere Exocortex renders it — driven entirely by my vault `exo__DisplayNameSpec` data, with no renderer-side hardcode — so the prefix set is a single, consistent, homoiconic source of truth.

## Statement (Gherkin)

```gherkin
Background:
  Given the vault ems__Task specs: 🔄 (status Doing), ✅ (status Done), … (from the shipped line)
    And a live 🚩-blocked exo__DisplayNameSpec on ems__Task (matchHostFunction isEffortBlocked) authored as part of THIS req

Scenario: a blocked Doing task composes 🚩 + 🔄 across all surfaces
  Given an ems__Task that is Doing AND blocked (its ems__Effort_blocker resolves to a not-done asset)
  When the plugin resolves its displayName (native link / DailyNote table / relation table / code-block table)
  Then it is "🚩 🔄 <label>" — both prefixes composed in the declared order (blocker before status)
    And the same result appears on EVERY surface (no surface shows only one marker)

Scenario: a blocked non-Doing task shows a single 🚩 — no double
  Given a blocked ems__Task that is NOT Doing
  When the plugin renders it in the DailyNote table (which today prepends a residual 🚩)
  Then it is "🚩 <label>" — exactly one 🚩 (the renderer residual is REMOVED; 🚩 comes only from the resolver)

Scenario: single participating spec composes to itself (no regression)
  Given a task with exactly one participating prefix spec (e.g. only 🔄 Doing, not blocked)
  When resolved
  Then it is "🔄 <label>" — byte-identical to the pre-composition single-template result

Scenario: no participating spec → plain label (no regression)
  Given a task with no participating spec
  When resolved
  Then it is "<label>"
```

## Verification

**Revert-verified (component, production-shape):** the bound suite (`@req:1a550210-f07e-4cbc-8707-6d4b428bba11`) drives the real `PrintNameRuleService.scanVault → DisplayNameResolver.resolve` over an in-memory vault carrying the 🔄-Doing + 🚩-blocked specs + a blocked-Doing task whose blocker resolves not-done — the real `isEffortBlocked` runs, composition produces "🚩 🔄 <label>".

Revert-verify axes (record in the PR body at implementation time):
- Revert the composition (back to single-winning-template) → the blocked-Doing task loses one of 🚩/🔄 → **RED**; restore → **GREEN** ("🚩 🔄 <label>").
- Revert the residual removal (re-add the renderer 🚩) with the live 🚩 spec present → a blocked non-Doing DailyNote row shows "🚩 🚩" → the no-double assertion **RED**; restore → **GREEN**.
- Single-spec + no-spec assertions GREEN both ways (guards no-regression).

## Design latitude / escalation

The composition MODEL is a real design: how a spec declares it contributes a composable PREFIX vs a full template; the compose ORDER (a spec order/priority field vs a matcher-kind ordering blocker→status→class); how the base label is chosen once. Model it minimally on the existing parts (PrintedLiteral prefix + PrintedProperty label) + priority. ⛤ **If this genuinely needs a new ontology / an onto-RFC (e.g. a new composition semantics on exo__DisplayNameSpec), ESCALATE (WAITING_DECISION) EARLY rather than over-building** — this is the most fundamental change of the line, and a design spike + escalation is preferable to a rushed model. The three deliverables (engine + live spec + residual removal) may split across PRs if that is cleaner (but the residual removal MUST land with the live spec to avoid a double-🚩 window).
