# req(exo): ExoSync makes pulled assets visible in Layouts without restart (disk-read reindex)

## Job Story

When ExoSync pulls a new or changed asset from the remote, I want it to appear in the auto-render Layout of its parent immediately, without reloading or restarting Obsidian, so multi-device sync (Desktop + iPhone) is not silently stale.

## Statement (Gherkin)

```gherkin
Given an Obsidian session whose triple store backs the auto-render Layout
When an ExoSync run pulls (or merges) at least one asset to local disk
Then those changed paths are targeted-reindexed straight from disk into the shared store
  And the synced asset is visible in a Layout-shaped SPARQL query without a restart
  And a pulled deletion removes the asset from the store the same way
  And a no-op / push-only sync triggers no reindex
  And the reindex reads frontmatter from disk, so a stale/empty metadataCache does not hide the asset
  And a reindex failure never mutates the sync outcome (best-effort, explicit Notice)
```

## Verification

**Revert-verified (integration):** `@req:e157d3e5-03b7-4e40-822a-3c946b72f32a` — neutering the `reindex` callback wired into the real `SyncCommands` (the post-sync targeted disk-read reindex) makes the «pulled asset visible in shared store» case go **RED** (the synced asset stays absent — the user-reported bug); restoring it → **GREEN**. The same suite covers the delete-case, the gate (no-op / push-only → 0 reindex), and the disk-read-over-stale-metadataCache case.

## Notes

Tier 1 of RFC `8f93ff95-c7c1-4d60-a2f4-7efc48272d10`. The reindex reads each changed path via `vault.adapter.read` (mobile-safe, no `Platform.isMobile` gate) and dispatches per-path by disk presence (present → targeted update, absent → triple removal), then runs a single `runInference()` + UI refresh bundle. ```exo-layout``` code-block visibility and full metadataCache decoupling are out of scope (Tier 2, D2/D3).
