# req(exo): instantiatePrototypeSubtree fails loud on an unresolvable node class + refuses duplicate re-deploy

## Job Story

When I deploy a WBS prototype subtree (e.g. the quarterly review-sheet for one employee × one quarter) with one click, I want the deployment to (1) refuse to write half-broken assets when a node's class can't be resolved, and (2) refuse to silently spawn a second identical subgraph if I accidentally click twice — so a mistake gives me a clear error instead of dangling `[[undefined]]` wikilinks (that redden SHACL later) or a duplicated review-sheet I have to hunt down and delete.

## Statement (Gherkin)

```gherkin
Scenario: M1 — fail loud on an unresolvable node class
  Given a prototype subtree in which one node's exo__Instance_class cannot be resolved to a class label
  When instantiatePrototypeSubtree runs on that subtree
  Then it throws a clear error naming the offending node
  And no instance file is written at all (validation is a pre-pass; zero partial state)
  And in particular no instance is written with a dangling exo__Instance_class wikilink to an unresolved class

Scenario: M2 — refuse to re-deploy the same subtree
  Given the vault already contains an instance whose exo__Asset_prototype is the root prototype
    And that instance's exo__Asset_relates set equals the current ref-params (same employee + quarter)
  When instantiatePrototypeSubtree runs again with those same ref-params
  Then it throws an explicit "already deployed" error naming the prototype and the params
  And no second subgraph is written

Scenario: M2 — a different employee/quarter is NOT blocked
  Given the vault already contains a deployment of the root prototype for employee A / quarter Q1
  When instantiatePrototypeSubtree runs for employee B (or quarter Q2) — a different ref-param set
  Then it deploys normally (the guard matches only on the exact prototype + ref-param set)
```

## Direction decision (approve gate)

**M1** direction is settled: **abort the whole deployment** (throw, mirroring the existing root-not-found guard) rather than skip-node-with-log — because skipping a node leaves dangling `ems__Effort_parent`/`ems__Effort_blocker` edges pointing at the omitted node (blocker re-map falls back to the ORIGINAL prototype). Fail-loud + atomic (pre-pass validation) is the clean choice.

**M2** direction — **recommended: explicit error** ("Ревьюшница `<params>` для этого прототипа уже развёрнута") rather than a silent no-op, so the accidental re-click is surfaced (ADHD-user operational safety), consistent with M1's fail-loud philosophy. A silent no-op would look like "nothing happened". **This is the product decision for the approve gate** — if Andrey prefers a silent no-op (idempotent re-run returns cleanly), the req narrows to "no second subgraph is written" without the error.

## Out of scope (ride-along refactor, no behavior change)

M4 (DRY): the file's local `extractRef` helper is replaced by importing the canonical `packages/core/src/utilities/extractAssetReference` (created specifically to de-triplicate this exact helper, audit #3384-H4). Behavior-preserving — verified by the existing test suite staying green. Not part of this requirement's behavior spec.
