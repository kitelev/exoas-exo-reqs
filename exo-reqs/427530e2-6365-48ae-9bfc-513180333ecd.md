# req(testsys): a NEW over-mock pass-through service-wrapper test fails the test-anti-pattern lint guard

## Job Story

When a contributor (human or AI) adds a NEW plugin test that fully mocks a central internal collaborator and asserts only that a method was called (pass-through delegation), I want the required `lint` CI job to fail with a pointer to the integration-lift policy — so that the Testing-Trophy opportunistic-lift shift (P3) is enforced by a ratchet, not by reviewer memory, and pure pass-through over-mock tests cannot creep back into the suite.

## Statement (Gherkin)

```gherkin
Given the test anti-pattern guard with the over-mock baseline pinned at 0
  (no pure pass-through over-mock file exists today — the canonical one,
   SPARQLQueryService.test.ts, was deleted and lifted to an integration suite)
When a NEW test file is added that wholesale-mocks a central collaborator
  (@kitelev/exocortex-core/-services, /services/, /adapters/, /infrastructure/,
   VaultRDFIndexer) AND asserts ONLY delegation (toHaveBeenCalled*) with no
   output-value assertion anywhere in the file
Then scripts/check-test-antipatterns.sh fails (exit 1) because the over-mock
  count rose above the baseline, pointing the author at TESTING.md
  §"Lift over-mock service-wrappers to integration"
And a file that mocks the same collaborator but also asserts an output value
  (toEqual/toBe/…) is NOT flagged (it exercises real behaviour, not pass-through)
And removing the offending over-mock file restores a green guard (exit 0)
```
