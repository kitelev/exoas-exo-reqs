# req(exo): generic cli set-property sets an arbitrary non-guarded frontmatter property on an existing asset

## Job Story

When I have watched a film and want to flip `concept__Movie_watched: false → true` on an existing Movie, or when I need to repoint `exo__Asset_isDefinedBy` on a task to move it between assetspaces per the co-location invariant, I want a single CLI call that sets the property — not a raw frontmatter `Edit` that bypasses the "maximum dogfooding — all mutations via CLI" workflow (and the `dogfood-cli-mutation` hook). At the same time, I do NOT want this generic primitive to become a way to bypass the guarded state-machine commands (status/zone/parent/label keep their dedicated commands with their preconditions), so those must be refused and routed to the right command.

## Statement (Gherkin)

```gherkin
Given an existing Movie asset with concept__Movie_watched: false
When I run `cli set-property <path> --input '{"property":"concept__Movie_watched","value":true}'`
Then the asset's concept__Movie_watched is true
And the asset's exo__Asset_updatedAt is bumped to the current timestamp

Given an existing task asset
When I run `cli set-property <path> --input '{"property":"ems__Effort_status","value":"[[<done-uid>]]"}'`
Then the command fails with an error naming the dedicated status command (apply mark-done / move-to-backlog / start-effort)
And the asset file is left unchanged (the guard did not write the status)

Given an existing task asset
When I run `cli set-property <path> --input '{"property":"ems__Task_zone","value":"[[<zone>]]"}'`  (or exo__Asset_label / ems__Effort_parent / ems__Effort_startTimestamp)
Then the command fails and names the dedicated command (set-criticality-* / set-label / set-parent / start-effort)
And the asset file is left unchanged

Given an existing asset
When I run `cli set-property <path> --input '{"property":"exo__Asset_uid","value":"..."}'` (or exo__Asset_updatedAt)
Then the command fails (identity property is immutable / updatedAt is auto-managed)
And the asset file is left unchanged

Given an existing task asset physically located outside its new anchor's ontology folder
When I run `cli set-property <path> --input '{"property":"exo__Asset_isDefinedBy","value":"[[<new-anchor>]]"}'`
Then exo__Asset_isDefinedBy is repointed to the new anchor
And exo__Asset_updatedAt is bumped
And a co-location warning naming `apply repair-folder` is emitted (the file still needs a physical move)

Given a wikilink value that does not resolve in the vault
When I run `cli set-property <path> --input '{"property":"exo__Asset_relates","value":"[[<missing-uid>]]"}'` without --skip-wikilink-validation
Then the command fails with a wikilink-not-found error
```

Reverting the property write (Edit-break the `updateProperty` call → the set-value assertion RED), the guard (Edit-break the denylist check → the guard-refuse assertion RED), the updatedAt bump, and the co-location check (each independently) makes exactly the corresponding assertion RED; restored → GREEN. The non-guarded happy path and the negative controls isolate non-vacuity.

## Homoiconicity Invariant decision (Q1/Q3) + rationale

**Chose option (b) — a generic CLI primitive — over (a) per-class homoiconic commands, for the open-ended "everything else" class of properties.** Homoiconicity Q1: user-configurable *semantics* (status state machine, criticality zones, parent/label with their preconditions) stay homoiconic and keep their dedicated `exocmd__Command` groundings — the `set-property` guard *enforces* that by refusing those properties. But an arbitrary scalar/boolean/enum/wikilink property (`concept__Movie_watched`, `exo__Asset_isDefinedBy`, any future non-command-backed property) is not a state machine — it is a low-level frontmatter mutation. Per Q3 this is a **core-of-processing** concern (a bare CLI verb, like `create`, that bypasses groundings), so a generic CLI primitive is the correct home; it subsumes the whole recurring "I had to raw-Edit a property" class (#3795) plus the `isDefinedBy` repoint (#3848) in one mechanism instead of proliferating one command per boolean/enum property.
