# req(exo): the Relations block unifies inline + reified relations with dedup (inline-wins) and provenance tagging

## Job Story

When I open an asset that has relations expressed both inline (frontmatter / backlinks)
and reified (`exo__Statement` instances), I want the Relations block to show each
logical relation exactly once — with reified ones attributed to their backing
statement and a duplicate inline+reified edge shown as the inline one — so that I see a
single, correctly-sourced relation list (the precondition for the "reified · &lt;AS&gt;"
marker and reify-editing) instead of a doubled or mis-attributed list.

## Statement (Gherkin)

```gherkin
Given an asset A with some inline relations (frontmatter backlinks) and some reified
  relations (exo__Statement instances, sourced via the Task 1.2 gated+whitelisted fetch)
When getAssetRelations(A) runs
Then the inline relations are tagged provenance:'inline'
  And each reified relation is appended tagged provenance:'reified' with its backing
      statement asset's path (statementPath), the OTHER end of the edge as the related
      asset, and the predicate in frontmatter-key form
  And an edge present BOTH inline and reified appears ONCE — as the inline one
      (inline-wins): the canonical (subject-uid, predicate-key, object-uid) key matches
      across both representations (both dual-IRI forms resolve to the same uid)
  And an outgoing reified edge (A is the statement subject) is additive (never deduped
      against an inline backlink)
  And a literal-object reified statement is skipped (a property value, not a relation)
  And reified relations surface even when A has zero inline backlinks
```
