# req(exocmd): prototype-class assets hide instance-lifecycle commands (homoiconic not-a-prototype precondition)

## Job Story

When I open a task/project/etc. **template** (an asset whose class is
`ems__TaskPrototype` / any `exo__Prototype` descendant), I want it to show only
the affordances that make sense for a template — instantiate it (Create
Instance) and universal maintenance (Clean Properties, Repair Folder, …) — and
NOT the instance-lifecycle buttons (Start, Mark Done, Set Criticality, status
transitions, child-creation, convert, vote), so that a template is never
mistaken for a live work-item.

## Statement (Gherkin)

```gherkin
Scenario: a prototype-template hides instance-lifecycle commands
  Given an asset whose exo__Instance_class is ems__TaskPrototype
    And ems__TaskPrototype IS-A ems__Task and IS-A exo__Prototype
  When the plugin resolves the asset's available commands
  Then "Start Effort", "Mark Done", "Set Criticality High" are NOT shown
   And "Clean Properties" (universal-maintenance) IS shown
   And the prototype's own "Create Instance" command IS shown

Scenario: a concrete instance keeps its lifecycle commands
  Given an asset whose exo__Instance_class is ems__Task (no prototype in its chain)
  When the plugin resolves the asset's available commands
  Then "Start Effort", "Mark Done" ARE shown (subject to their status gate)

Scenario: status gate is preserved (no regression)
  Given a concrete ems__Task with no status set
  When the plugin resolves "Start Effort" availability
  Then it is NOT shown (the existing status precondition still applies)
```

## Mechanism

- **Reusable clause (shipped):** `ASK { FILTER NOT EXISTS { $target exo:Instance_class ?c . FILTER(STRENDS(STR(?c), "Prototype")) } }`.
  A suffix check on the asset's symbolic `exo:Instance_class` IRI — true iff the
  asset's class is named `…Prototype`. **NOT** the originally-designed transitive
  `exo:Class_superClass*` walk: that is broken on the real store (the dual-IRI
  gap — see CLAUSE FINDING — `instance_class` resolves to the symbolic class IRI
  while `Class_superClass` lives on the file IRI, so SPARQL can't walk it).
- **Homoiconic, no engine code.** `PreconditionEvaluator` already supports
  `exocmd__Precondition_sparqlAsk` (`STRENDS`/`STR` + `FILTER NOT EXISTS`,
  empirically verified against the real store). The clause is conjoined into each
  lifecycle command's command-level precondition; universal-maintenance +
  create-instance commands carry no such clause.
- **Tiers** (do NOT carry the clause): universal-maintenance (targetClass
  `exo__Asset`, ×6) and prototype create-instance (targetClass a prototype-class,
  ×8). Meta commands (Create Class/Instance/Subclass/Narrower Concept) target
  `exo__Class`/concept and never appear on a prototype-template asset.

## Notes

- **Revert-verifiable.** Dropping the not-a-prototype FILTER from a lifecycle
  precondition makes a prototype resolve Start (RED); restoring hides it (GREEN).
- **Forget-leak guarded.** A parity test over the real `exoas-exocmd` submodule
  fails CI if a future instance-lifecycle command ships without the clause.
