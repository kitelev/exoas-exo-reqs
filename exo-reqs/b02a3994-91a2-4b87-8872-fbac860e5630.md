# req(exo): a git-free vault carries no `.gitmodules`; readers re-derive URLs via deriveUrl (RFC 0005 Phase 1)

## Job Story

When I mount AssetSpaces on a git-free vault (iPhone, and post-#3567 every canonical desktop vault is git-free too), I want the vault to carry **no `.gitmodules` sidecar at all** — a git-named file on a non-git vault is confusing and serves no purpose — while every device-side rediscovery flow (unmount listing, clone-needs-fetch re-fetch, apply-profile switch-back) keeps working by reconstructing each AssetSpace's GitHub URL from its on-disk folder path (`deriveUrl`), so the URL never has to be stored in a `.gitmodules` registry.

## Statement (Gherkin)

```gherkin
Scenario: mounting on a git-free vault writes no .gitmodules
  Given a git-free vault with no .gitmodules file
  When an AssetSpace is mounted via the REST path (bootstrap, add-assetspace, or apply-profile)
  Then the AssetSpace's content is materialised under assetspaces/<owner>/<repo>
  And no .gitmodules file is created or written

Scenario: the unmount listing reconstructs the mounted set from the filesystem
  Given a git-free vault with AssetSpaces materialised at assetspaces/<owner>/<repo> and no .gitmodules
  When the «Unmount assetspace» command lists what is mounted
  Then it lists each materialised assetspaces/<owner>/<repo> folder
  And each entry's URL is deriveUrl(folderPath) = https://github.com/<owner>/<repo>

Scenario: clone-needs-fetch is detected and re-fetched from the filesystem, not .gitmodules
  Given a git-free vault whose assetspaces/<owner>/<repo> folders exist but are empty (synced without content) and no .gitmodules
  When the Bootstrap command inspects the vault
  Then it classifies the state as clone-needs-fetch from the present-but-empty folders
  And re-materialises each from deriveUrl(folderPath), not from a .gitmodules URL entry

Scenario: a non-conforming folder is skipped, not spliced into a URL
  Given a folder under assetspaces/ that is not a canonical assetspaces/<owner>/<repo> (one level, or an out-of-charset / traversal segment)
  When the readers enumerate the filesystem
  Then deriveUrl returns null for that folder
  And the folder is skipped (no malformed URL is produced)

Scenario: the desktop git-submodule path is unchanged (out of scope)
  Given a legacy git-superproject vault with a committed .gitmodules and real gitlinks
  When apply-profile runs on the git-submodule path
  Then it still reads .gitmodules (committed in the repo) — Phase 1 does not touch git vaults (RFC 0005 Phase 2 is N/A)
```

## Verification (plan)

**Binding class:** `Integration` (a behaviour change spanning the REST mount writer + two readers; above a pure-unit binding). Test title carries `@req:b02a3994-91a2-4b87-8872-fbac860e5630`.

**Revert-verify** ([[integration-test-revert-verify]]): the test exercises (a) a REST mount produces **no** `.gitmodules` entry, and (b) each reader reconstructs the mounted set from the filesystem with `deriveUrl`-derived URLs. Re-instating the `appendGitmodulesEntry` write (or re-pointing a reader at `.gitmodules`) in a type-preserving way makes the cited test **RED**; restored → **GREEN**.

## RFC §10.3 — does filesystem enum + Profile/registry fully replace the `.gitmodules` path-list?

Verified against origin/main (`85c338f5`, 2026-06-27), this is **yes**:

- **apply-profile (REST)** already computes "what's mounted" from filesystem presence (`derivePhysicallyMaterializedAsUids`) and sources URLs from descriptors — never from `.gitmodules`. So the most load-bearing flow is already replaced.
- **unmount listing** lists mounted folders — which are present on disk by definition of "mounted" — so a 2-level filesystem enum + `deriveUrl` fully covers it.
- **clone-needs-fetch** present-but-empty `assetspaces/<owner>/<repo>` folders are filesystem-detectable; fully-absent AssetSpaces are re-materialised by apply-profile (descriptor-driven, already filesystem-based). The "`.gitmodules` entries arrive without the folders" state is fundamentally a **git-clone artifact** (committed `.gitmodules` + `--no-recurse-submodules`); on a git-free vault `.gitmodules` is device-local and not synced by ExoSync (0 references), so that state does not arise.

**The one residual edge (the open decision):** a **legacy flat / un-derivable** mount (`assetspaces/exo`, pre-#3538) yields `deriveUrl → null` and would no longer be listed/re-fetched by the readers. Full-removal (Option A) accepts this (flat mounts are deprecated; canonical is 2-level Maven). A transition fallback (Option B) would additionally merge any pre-existing `.gitmodules` entries during a soak period. This is the transition-fallback decision surfaced to Andrey.

> RFC 0005 §6 Phase 1. Consumes the Phase 0 primitive `deriveUrl` (req `[[1e56367b-ca1d-4c8b-8839-343e82cee2d3]]`, Active). Phase 2 is N/A (no git-superproject vault remains in active use).
