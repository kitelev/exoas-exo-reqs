# req(exo): Un-archive Ontologically — the ontological inverse of «Archive Ontologically»

> Source: session 2026-07-12 (Andrey: «есть un-archive, но нет ontologically деархивации — почему?»). Investigation showed the asymmetry is by-construction: the flag-archive layer («Archive Completed» 52c88678 / «Un-archive» 59aaf685) is losslessly reversible (add/remove one `archived` flag, no file move), while «Archive Ontologically» (3de74a87) OVERWRITES `exo__Asset_isDefinedBy` + relocates the file, storing no original. So the flag layer has an inverse and the ontology layer does not. This req adds the missing ontological inverse.
>
> **Design decided (Andrey, Option A):** recover the source ontology by REVERSING the dedicated `exo__Ontology_archiveOntology` relation — the symmetric inverse of the forward «Resolve archive ontology» NamedQuery (5e2210f4). No change to the forward command; works for all already-archived assets; correct under the documented 1:1 archive-pairing pattern (concept e15d22ae).

## Job Story

When I ontologically-archived an asset by mistake (or its context changed), I want a single «Un-archive Ontologically» button that puts it back into its original ontology and folder — the exact inverse of «Archive Ontologically» — driven entirely by my vault archive-pairing data, with no renderer/engine hardcode.

## Statement (Gherkin)

```gherkin
Background:
  Given a source ontology $S that declares exo__Ontology_archiveOntology → archive ontology $A (1:1 pairing)
    And an asset X that was ontologically-archived (its exo__Asset_isDefinedBy = $A, file in the $A folder)
    And the homoiconic «Un-archive Ontologically» command bound to exo__Asset

Scenario: Un-archive Ontologically restores the asset to its source ontology and folder
  Given the ontologically-archived asset X (isDefinedBy = $A)
  When I run «Un-archive Ontologically» on X (apply un-archive-ontologically X)
  Then X.exo__Asset_isDefinedBy is re-anchored to $S (the source ontology, found by reversing Ontology_archiveOntology)
    And X is physically relocated into the $S folder (co-location)

Scenario: the command is visible ONLY on an ontologically-archived asset
  Given a NORMAL asset Y whose isDefinedBy = $S (a source ontology, not anyone's archiveOntology)
  When resolve-buttons runs on Y
  Then «Un-archive Ontologically» is NOT offered (Y is not ontologically archived)
    And on the ontologically-archived asset X the command IS offered

Scenario: idempotency — no double un-archive
  Given asset X already restored (isDefinedBy = $S)
  When «Un-archive Ontologically» is applied again
  Then it is a no-op (the precondition fails; isDefinedBy is a source ontology, nobody's archive target)

Scenario: round-trip is identity
  Given an active asset Z (isDefinedBy = $S, file in the $S folder)
  When «Archive Ontologically» then «Un-archive Ontologically» are applied to Z
  Then Z.isDefinedBy = $S again and Z is back in the $S folder (identity on isDefinedBy + folder)
```

## Design notes

- **Reverse mechanism (Option A):** `$currentAsset exo:Asset_isDefinedBy ?arch . ?src exo:Ontology_archiveOntology ?arch` → `?src` = source ontology. Symmetric to forward NamedQuery 5e2210f4. Uses the purpose-built relation reversed (cleaner than reversing the general `Ontology_imports`, which is multi-valued).
- **Injectivity:** correct iff each archive ontology is the `archiveOntology` of exactly one source (the documented 1:1 pairing, concept e15d22ae; verified on current data — 1 declaration, 1:1). If violated, the reverse is ambiguous → the command must NOT fire (single-`?src` guard in the precondition / a scalar property_set that fails loudly on multiple).
- **Zero engine code expected** (homoiconic-gap-triage): composite + property_set + service_call repairFolder + query__NamedQuery all exist; the forward command (3de74a87 / composite ee48c626) is the exact template. If a primitive is genuinely missing → file a dogfood-gap issue, do NOT hand-roll engine code.
- **Assetspace:** ship to the exocmd command assetspace (where 3de74a87 + its composite live), co-located by isDefinedBy.
