# req(exo): plugin-settings live-mirror is a read-only transitional watcher (M2.3)

## Job Story

When I have homoiconic plugin settings enabled, I want vault `exo__Setting` asset
edits to keep flowing into the UI (asset→UI), but I do NOT want every UI toggle
silently auto-written back into a vault asset — the RDF asset should be the
canonical authority and I persist my UI changes explicitly via «Export settings»
— so settings distribution is one deliberate mechanism (Export/Import), not a
continuous bidirectional mirror, during the deprecation window before the
live-mirror is removed.

## Statement (Gherkin)

```gherkin
Given the homoiconic settings store is active (read-only transitional, M2.3)
When I change a setting in the UI (saveSettings → pushChangedFields)
Then the vault asset is NOT auto-rewritten (the continuous UI→asset auto-write is deprecated)
And a one-time deprecation notice is logged directing me to «Export settings»

Given the read-only transitional store
When a setting asset changes (a vault edit / a synced cross-device change arrives)
Then the inbound watcher still applies it to the UI (asset→UI, canonical = RDF)

Given a setting the user toggled in the brief onload→scan window (a "dirty" field)
When the initial scan overlay runs
Then the dirty field is NOT auto-pushed to the asset (no outbound write) — the user persists it via «Export settings»
```
