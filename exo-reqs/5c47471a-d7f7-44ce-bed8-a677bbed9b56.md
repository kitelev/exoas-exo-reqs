# req(exo): both `$today` resolution paths return today's LOCAL calendar day (not UTC)

## Job Story

When a `$today` PropertyDefault / body-template token is resolved just after local midnight (my timezone is UTC+5, so UTC is still the previous calendar day), I want it to resolve to **today's LOCAL date** — matching what `$date` already returns and what a human reading their calendar sees — so `$today`-defaulted properties are never stamped with yesterday's date, and `$today` never disagrees with `$date` at the local/UTC day boundary.

## Statement (Gherkin)

```gherkin
Scenario: $today just after local midnight in a UTC+N timezone returns the LOCAL day
  Given the process timezone is Asia/Almaty (UTC+5, no DST)
  And the wall clock is 2026-07-02T19:27:00Z (= 2026-07-03T00:27 local, local day 03, UTC day 02)
  When a $today SubstitutionToken PropertyDefault is resolved by the parse-time resolver
  Then the resolved value is "2026-07-03" (today's LOCAL calendar day)
  And it is NOT "2026-07-02" (the UTC day, which the former UTC form produced)

Scenario: registry $today matches $date at the boundary
  Given the same instant
  When both the registry $today resolver and the registry $date resolver resolve
  Then both return "2026-07-03" (they agree on today's LOCAL calendar day)
```

## Verification (plan)

**Binding class:** `Unit` — the fix is a leaf arithmetic change in two resolvers. Parse-time is exercised production-shape through the real loader `CommandResolver.loadCommand` (a real `InMemoryTripleStore` seeds a `$today` token + PropertyDefault + create_instance grounding, and the resolved PropertyDefault value is asserted); the registry path is exercised via `getResolver("today")`. Test title carries `@req:5c47471a-d7f7-44ce-bed8-a677bbed9b56`.

**Revert-verify** ([[integration-test-revert-verify]]): with either `$today` path reverted to `toISOString().slice(0,10)`, the test resolves to `"2026-07-02"` → **RED** (asserts `"2026-07-03"`); restored to the local form → **GREEN**. CI-robustness ([[jest-timezone-sensitive-tests]]): `process.env.TZ` cannot be re-tzset at runtime under jest (V8 caches the worker timezone), and the fix has NO observable effect in a UTC runner — so the test installs a `Date` subclass (`FakeAlmatyDate`) simulating a fixed UTC+5 (Asia/Almaty, no DST) offset at the instant `2026-07-02T19:27:00Z` (= 00:27 local, local day 03 / UTC day 02), independent of the runner's real timezone. A guard (`getHours()===0`, `getUTCDate()===2`) asserts the simulated tz is active so the test can never silently pass both ways in a UTC-tz CI runner.
