# req(testsys): jest coverage thresholds are monotonic — a downward drift fails the lint guard

## Job Story

When a contributor lowers a jest `coverageThreshold.global` metric to make a red coverage build pass — without consciously lowering the committed baseline — I want the required `lint` CI job to fail, so that a coverage regression can never silently drift down as a config tweak; lowering the high-water mark must be an explicit, req-justified act.

## Statement (Gherkin)

```gherkin
Given a committed coverage high-water mark in scripts/coverage-thresholds-baseline.json
  (e.g. obsidian-plugin branches >= 63, core branches >= 95)
When a jest.config.js coverageThreshold.global metric is lowered BELOW its
  baseline value (without lowering the baseline number too)
Then scripts/check-coverage-monotonic.mjs fails (exit 1) reporting the metric
  that dropped below baseline
And raising the threshold back to (or above) the baseline restores a green guard
And growing a threshold above its baseline is always allowed (no failure)
```
