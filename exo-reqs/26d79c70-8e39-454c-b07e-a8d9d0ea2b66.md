# req(exo): `$today` executor + form-prefill surfaces return today's LOCAL calendar day (not UTC)

## Job Story

When a create-instance grounding stamps `$today` into a label / property / nominal instance date, OR the create-instance modal pre-fills its planned-date field with `$today`, just after local midnight (my timezone is UTC+5, so UTC is still the previous calendar day), I want both to resolve to **today's LOCAL date** — matching the already-local `$date`/`$tomorrow`/parse-time+registry `$today` tokens and what a human reading their calendar sees — so a newly-created asset is never stamped/prefilled with yesterday's date and the four `$today` surfaces never disagree at the local/UTC day boundary.

## Statement (Gherkin)

```gherkin
Scenario: executor $today just after local midnight in a UTC+N timezone returns the LOCAL day
  Given the process timezone is Asia/Almaty (UTC+5, no DST)
  And the wall clock is 2026-07-02T19:27:00Z (= 2026-07-03T00:27 local, local day 03, UTC day 02)
  When GroundingExecutor.substituteVariables interpolates "$today" into a template
  Then the interpolated value is "2026-07-03" (today's LOCAL calendar day)
  And it is NOT "2026-07-02" (the UTC day, which the former UTC form produced)

Scenario: form $today prefill returns the LOCAL day at the boundary
  Given the same instant
  When the DynamicForm resolves a date field's "$today" defaultValue
  Then the prefilled value is "2026-07-03" (today's LOCAL calendar day)
  And it is NOT "2026-07-02" (the UTC day)
```

## Verification (plan)

**Binding class:** `Unit` — the fix is a leaf change in two date-slicing sites (executor `resolveInstanceDate` + form `resolveDefaultValue`). The executor path is exercised through the real `GroundingExecutor.substituteVariables`; the form path through the real `DynamicForm` render (jsdom). Test titles carry `@req:26d79c70-8e39-454c-b07e-a8d9d0ea2b66`.

**Revert-verify** ([[integration-test-revert-verify]]): with either surface reverted to `toISOString().slice(0,10)`, the boundary test resolves to `"2026-07-02"` → **RED** (asserts `"2026-07-03"`); restored to the local form → **GREEN**. CI-robustness ([[jest-timezone-sensitive-tests]]): `process.env.TZ` cannot be re-tzset at runtime under jest (V8 caches the worker timezone), and the fix has NO observable effect in a UTC runner — so both tests install a `Date` subclass (shared `installFakeOffsetDate` helper) simulating a fixed UTC+5 (Asia/Almaty, no DST) offset at the instant `2026-07-02T19:27:00Z` (= 00:27 local, local day 03 / UTC day 02), independent of the runner's real timezone. A guard (`getHours()===0`, `getUTCDate()===2`) asserts the simulated tz is active so the tests can never silently pass both ways in a UTC-tz CI runner.
