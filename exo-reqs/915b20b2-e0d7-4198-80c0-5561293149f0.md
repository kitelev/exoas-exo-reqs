# req(exo): ems__WaitingCheckTask — iterative «Waiting For» check task + prevIteration chain + next-iteration button

## Job Story

When I am waiting for an external answer (e.g. a consultant's reply on my residence-permit project) and there is nothing to DO yet except «check again tomorrow», I want a first-class task that, on the day it is planned, offers a one-tap **«Следующая итерация»** — closing today's check as Done and spawning tomorrow's identical check (same context, body, parent, zone) planned for the next day — so that the waiting stays visible as a soft daily tickler without me re-typing it, and when the answer finally arrives I just mark it done normally; afterwards I can see how many checks it took by the length of the `prevIteration` chain.

## Statement (Gherkin)

```gherkin
Scenario: subclass inherits the Task workflow (standard mark-done works on a WaitingCheckTask)
  Given an ems__WaitingCheckTask instance (exo__Class_superClass ⊒ ems__Task) with ems__Effort_status = Doing
  When the standard mark-done workflow_transition grounding is applied to it
  Then WorkflowResolver resolves the built-in Task workflow for it (via the superClass walk)
  And ems__Effort_status becomes Done

Scenario: next-iteration button clones the current check into the next one
  Given an ems__WaitingCheckTask "Проверить ответ консультанта" with ems__Task_zone Z, ems__Effort_parent P, exo__Asset_relates R, and a non-empty body
  When the «Следующая итерация» command is applied to it
  Then a NEW ems__WaitingCheckTask is created with the same exo__Asset_label, the same ems__Task_zone Z, ems__Effort_parent P, exo__Asset_relates R, and the same body
  And the new instance has a fresh exo__Asset_uid and exo__Asset_createdAt
  And the new instance has ems__Effort_status = Backlog
  And the new instance has ems__Effort_plannedStartTimestamp = tomorrow's date
  And the new instance has ems__Effort_prevIteration pointing at the original check
  And the new instance carries NO start/end/resolution timestamp
  And the ORIGINAL check now has ems__Effort_status = Done

Scenario: chain end — ordinary mark-done, no next iteration
  Given an ems__WaitingCheckTask that is the current tail of a prevIteration chain
  When the user marks it done normally (not the «Следующая итерация» button)
  Then it becomes Done and NO next iteration is created
  And the number of checks it took = 1 + the count of ems__Effort_prevIteration hops back from it

Scenario: body-clone no-op — empty body stays empty (no regression)
  Given an ems__WaitingCheckTask with an EMPTY body
  When «Следующая итерация» is applied
  Then the new instance is created with an empty body (no spurious content)
  And every other clone/tickler assertion still holds
```
