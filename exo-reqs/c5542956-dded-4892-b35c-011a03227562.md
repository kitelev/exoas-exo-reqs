# req(exo): exo-layout action buttons are precondition-aware (#3654 Part 1)

## Job Story

When I author an `exo-layout` table with action buttons whose visibility should depend on the row asset's state (e.g. a "Start" button only on tasks that are not already started), I want each button gated by its `exo__Precondition_sparql` ASK against the live graph, so that buttons which don't apply to a row are hidden instead of always showing — matching how structural exocmd command buttons already gate via `PreconditionEvaluator`.

## Statement (Gherkin)

```gherkin
Scenario: an action button whose precondition ASK evaluates FALSE is hidden
  Given an exo-layout table row for a real vault asset
  And an action button carrying an exo__Precondition_sparql ASK that does NOT hold for that asset
  When the layout renders and the precondition is checked against the live graph
  Then the button is hidden

Scenario: an action button whose precondition ASK evaluates TRUE is shown
  Given an exo-layout table row for a real vault asset
  And an action button carrying an exo__Precondition_sparql ASK that DOES hold for that asset
  When the layout renders and the precondition is checked against the live graph
  Then the button is shown

Scenario: the $target placeholder resolves to the store's real subject IRI (no mis-gating)
  Given an action button precondition ASK that references $target
  When the precondition is checked for a row whose asset path contains slashes
  Then $target is substituted with the asset's actual store subject IRI (obsidian://vault/<encodeURI path>)
  And the ASK evaluates against the asset that really exists in the graph (not an over-encoded non-existent subject)
```

## Verification (plan)

**Binding class:** `Integration` — the behavior spans `SPARQLApi.ask` + the engine (real parse/translate/executeAsk against a real in-memory triple store) + `LayoutCodeBlockProcessor` wiring + `$target` substitution. Test titles carry `@req:c5542956-dded-4892-b35c-011a03227562`.
