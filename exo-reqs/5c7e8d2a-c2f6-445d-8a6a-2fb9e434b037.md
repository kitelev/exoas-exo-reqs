# req(exo): cross-path same-uid keep-REMOTE / merged convergence (#3731)

## Job Story

When an incomplete co-location migration leaves the SAME asset at a flat path
locally and a co-located path on the remote (a cross-path same-uid conflict),
and I resolve it **keep-remote** (I want the remote's co-located version) or
**merged**, I want the resolution to STICK — the asset converges to one path,
the stale copy is removed, and the conflict never re-derives on the next sync —
so that resolving the conflict actually ends it (the keep-local symmetry of
#3729), instead of the conflict returning every sync.

## Statement (Gherkin)

```gherkin
Scenario: keep-remote of a cross-path same-uid co-located rename converges
  Given an asset's exo__Asset_uid lives at a FLAT path locally
    And the SAME uid lives at a CO-LOCATED path on the remote (no common base)
    And ExoSync has quarantined the cross-path conflict (the flat path surfaced)
  When I resolve the conflict choosing "remote"
  Then the resolver writes the remote content to the flat path, backs up the
       discarded flat-local copy, snaps the flat watermark base to the remote
       blob and unpins it — with NO remote commit
  And on the next sync the engine applies the resulting clean remote-rename:
       the stale flat-local copy is deleted and the co-located version is pulled
  And the asset exists on disk ONLY at the co-located path
  And no conflict re-derives on any subsequent sync

Scenario: merged of a cross-path same-uid co-located rename converges
  Given the same cross-path same-uid quarantined conflict
  When I resolve it choosing "merged" with merged content
  Then the deferred push consolidates: the merged content is pushed to the flat
       path AND the stale co-located remote sibling is deleted in one commit
  And no conflict re-derives on any subsequent sync
```

## Notes

- **#3731 was a phantom gap.** It was filed as a speculative follow-up to #3729
  assuming keep-remote shared the keep-local gap. Empirical verification
  (three production-shape tests, 2026-06-23) showed keep-remote **already
  converges** via the pre-existing snap-to-remote watermark mechanism, and
  merged already converges via the #3729/#3732 consolidation path. The
  asymmetry: keep-local/merged need a remote mutation (push + delete the stale
  remote sibling) that the per-path TOCTOU was dropping pre-#3729; keep-remote
  needs NO remote mutation (the remote is already canonical), so its
  convergence falls out of the existing snap→rename for free.
- **No production code changed.** The resolution of #3731 is this requirement
  plus the regression tests (which the investigation flagged as missing —
  cross-path same-uid was untested for keep-remote/merged), revert-verified
  against the snap mechanism so they cannot pass vacuously.
