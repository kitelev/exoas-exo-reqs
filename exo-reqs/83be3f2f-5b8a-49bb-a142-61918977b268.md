# req(exo): displayName resolver considers ALL instance classes (class-membership), not only the first

> Source: issue #3864, surfaced by req `[[a577316f-f2ff-469b-a6d3-ff946eccc68f]]` (PR #3862) + the ✅👥 / 👥 Meeting specs (exoas-exo `006b15c`). `DisplayNameResolver.extractAssetClass` uses `exo__Instance_class[0]` only. Real meetings are inconsistently classed: some `[[ems__Meeting]]` (Meeting-first → works) and some `[[ems__Task]], [[ems__Meeting]]` (Task-first → the resolver selects an `ems__Task` spec and never consults the `ems__Meeting` 👥 spec → the class prefix is dropped). Teach the resolver a class-MEMBERSHIP test so a class-level spec applies whenever its class is present.

## Job Story

When a note carries several `exo__Instance_class` values (e.g. a meeting that is both `ems__Task` and `ems__Meeting`), I want its displayName to reflect ANY class I have a spec for — so a 👥 (or ✅👥) prefix I declared for `ems__Meeting` appears regardless of which class happens to be listed first.

## Statement (Gherkin)

```gherkin
Background:
  Given a vault exo__DisplayNameSpec for ems__Meeting → "👥 " (unconditional, appliesToClass ems__Meeting)
    And the existing ems__Task conditional specs (🔄 Doing etc.)

Scenario: a Task-first multi-class meeting gets the class prefix (the reported bug)
  Given a note whose exo__Instance_class is [ [[ems__Task]], [[ems__Meeting]] ] (Meeting NOT first)
    And its status is NOT Doing (no conditional Task spec participates)
  When the plugin resolves its displayName
  Then it is "👥 <label>" — the ems__Meeting class spec applies because the class is PRESENT
    And today (first-class-only) it would be "<label>" (👥 dropped)

Scenario: highest-priority applicable spec wins across all classes
  Given the note is [ [[ems__Task]], [[ems__Meeting]] ] and the composed ems__Meeting+Done → "✅👥 " spec exists
    And the note's status is Done
  When the plugin resolves its displayName
  Then the highest-priority participating spec across ALL its classes wins (the composed ✅👥 spec)

Scenario: no regression for single-class notes
  Given a note with a single exo__Instance_class
  When the plugin resolves its displayName
  Then selection is byte-identical to today (the first-class and all-classes paths coincide)

Scenario: two distinct-prefix specs simultaneously applicable → ONE wins (composition out of scope)
  Given a Doing meeting where BOTH a 🔄-Doing (ems__Task) spec AND a 👥 (ems__Meeting) spec apply
  When the plugin resolves its displayName
  Then exactly ONE prefix is shown (the highest-priority spec) — this req does NOT compose 🔄+👥
```

## Verification

**Revert-verified (component, production-shape):** the bound suite (`@req:83be3f2f-5b8a-49bb-a142-61918977b268`) drives the real `PrintNameRuleService.scanVault → getTemplateForClass(metadata) → DisplayNameResolver.resolve` path over an in-memory vault carrying an `ems__Meeting → 👥` spec + a Task-first multi-class note — no hand-injected class.

Revert-verify axes (record in the PR body at implementation time):
- Revert the multi-class membership (resolver back to `instance_class[0]`) → the Task-first meeting loses "👥 " → assertion **RED**; restore → **GREEN** ("👥 <label>").
- Single-class note assertion stays GREEN both ways (guards no-regression).

## Out of scope (tracked)

- **Prefix composition** of two simultaneously-applicable distinct-prefix specs (🔄 Doing + 👥 Meeting → "🔄👥") — a further engine design; note it, do not build here.
- **#3865** — the computed 🚩 blocked matcher (separate matcher-engine extension).
