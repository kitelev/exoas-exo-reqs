# req(exo): `PreconditionEvaluator` `askCache` must not freeze compiled `$today`/date tokens across a local calendar-day change

## Job Story

When a dynamic command's visibility/availability precondition compares an asset's date against `$today` (or any calendar-date token) in a precondition-SPARQL ASK, and my Obsidian process has been open across local midnight, I want the precondition to be re-evaluated against **today's** local date automatically — without a manual store refresh / `invalidateCache` — so a button's visibility never mis-fires because a stale compiled ASK from *yesterday* is still cached.

## Statement (Gherkin)

```gherkin
Scenario: a $today precondition ASK compiled on day D recompiles automatically on day D+1 without manual invalidateCache
  Given a PreconditionEvaluator instance and an asset whose planned date is the LOCAL day "2026-07-04"
  And a precondition ASK: ASK { <asset> ems:plannedDate ?d . FILTER(?d = $today) }
  And the local calendar day is "2026-07-03" (asset is tomorrow)
  When PreconditionEvaluator.evaluate runs the precondition (compiling + caching the ASK)
  Then the ASK does NOT match (the asset is not today)
  When the local calendar day advances to "2026-07-04" (asset is now today) with NO invalidateCache call
  And PreconditionEvaluator.evaluate runs the SAME precondition on the SAME evaluator instance
  Then the ASK MATCHES today's LOCAL day (the command becomes available)
  And with the raw-text (non-date-aware) cache key the stale day-"2026-07-03" ASK would still be served → the ASK would NOT match (RED)

Scenario: a token-free precondition ASK is compiled ONCE and reused across local days (no perf regression)
  Given a precondition ASK containing NO calendar-date token
  When PreconditionEvaluator.evaluate runs it on local day D and again on local day D+1
  Then the ASK is compiled exactly once (the compiled entry is reused across the day change)
```

## Verification (plan)

**Binding class:** `Unit` — the fix is a leaf change confined to `PreconditionEvaluator` (`evaluateSparqlAsk` cache key + a temporal-token predicate). Both the cross-midnight recompile (production-shape `evaluate()` ASK through the real evaluator + `InMemoryTripleStore`) and the token-free perf-neutrality (compile-count via a spy on the real SPARQL parser) are exercised without mocking the date basis. Test titles carry `@req:96be4042-6521-4f62-bdc6-ee31e7e260a5`.

**Revert-verify** ([[integration-test-revert-verify]]): with the cache key reverted to the raw `sparqlAsk` text, the cross-midnight test re-serves the stale day-D compiled ASK → the day-D+1 evaluation returns the wrong boolean (asset not matched) → **RED**; the date-aware key → cache miss on the new day → recompile → correct match → **GREEN**. CI-robustness ([[jest-timezone-sensitive-tests]]): `process.env.TZ` cannot be re-tzset under jest (V8 caches the worker timezone) and the local/UTC date distinction has no observable effect in a UTC runner — so the test uses the shared `installFakeOffsetDate` `Date`-subclass helper to simulate a fixed UTC+5 (Asia/Almaty, no DST) offset, switching the fixed instant across the local-midnight boundary; a guard (`getHours()`/`getUTCDate()`) asserts the simulated tz is active so the test can never silently pass both ways.

**Empirical double-gate:** an ephemeral node script runs the REAL built `dist` `PreconditionEvaluator` with an injected `IClock` flipped from day D to day D+1 and asserts the same precondition flips from not-available to available across the day change (deleted after).

> 6th and final known surface of the date-tz family. Follow-up to `ecb90c06` (`$today` in `substituteVariables`, #3811/#3817); siblings `26d79c70` (#3809), `5c47471a` (#3808), `bca93bfb` (#3806). Design (Homoiconicity Invariant): the token default is vault-declared RDF (the precondition asset's SPARQL); only the domain-agnostic cache-keying / date arithmetic lives in code. No hardcoded timezone — the local day derives from `DateFormatter.toDateString(clock.now())`, sharing the `$today`-family basis.
