# req(exo): plugin «Register for sync» — write an exo__AssetSpace descriptor so a mounted pack enters the ExoSync materialized set

## Job Story

When I add a knowledge pack by URL and then realise my edits in it are not syncing (it has no descriptor of its own), I want to run «Register for sync» — on my iPhone just as on desktop — pick that pack, and have a proper exo__AssetSpace descriptor written into it so ExoSync starts tracking it; and I want to be told honestly whether it will push (I own it) or only pull (I don't), so I never lose work assuming a round-trip that can't happen.

## Statement (Gherkin)

```gherkin
Given a knowledge pack mounted at assetspaces/<owner>/<repo> with content but NO exo__AssetSpace descriptor
Then ExoSync's class-based discovery does NOT enumerate it into the materialized sync set
And it is surfaced only as a mounted-but-undeclared pack

When I run «Register for sync» and select that pack
Then a UID-named exo__AssetSpace descriptor is written into the pack folder via vault.adapter
And the descriptor carries exo__Instance_class exo__AssetSpace, exo__AssetSpace_source = https://github.com/<owner>/<repo> (derived from the mount path), exo__AssetSpace_namespace = <repo> minus the `exoas-` prefix, and a self-anchor exo__Asset_isDefinedBy
And ExoSync's class-based discovery now enumerates the pack into the materialized sync set (the «won't sync» state clears)

Given the PAT has push access to the pack's repo
Then the user is told edits will pull AND push

Given the PAT has no push access to the pack's repo
Then the pack is registered pull-only and the user is warned it will not push (no false push claim)

Given the plugin is loaded on mobile (no Node fs / git)
Then the «Register for sync» command is still registered (Desktop↔Mobile Command Parity) and writes the descriptor via vault.adapter
```
