# req(exo): read-view Relations block → block-level «Edit relations» affordance (desktop + mobile)

## Job Story

When I am viewing an asset and see its Asset Relations block, I want a visible «Edit relations» control right there so I can open the relations editor (list / create / delete / reify-toggle) without having to know the command-palette name «Edit properties» — so relation editing is discoverable from the read view, on my iPhone just as on desktop.

## Statement (Gherkin)

```gherkin
Given the read-view Asset Relations block of an asset, rendered with the asset's currentFile
Then a single block-level «Edit relations» affordance (.exocortex-edit-relations) is rendered

Given the read-view Asset Relations block with the «Edit relations» affordance
When I activate it
Then the current asset's property-editor (the Relations section reachable from the palette) opens
And it opens via the registered `exocortex:edit-properties` command (one source of truth — the same flow as Cmd+P and the ribbon)

Given the plugin is loaded on mobile
Then the affordance is wired with no platform gate, using only Obsidian-core APIs (no Node fs)
And activating it opens the editor exactly as on desktop (Desktop↔Mobile Command Parity invariant)

Given a legacy caller that does not thread the asset's currentFile into RelationsRenderer.render
Then no «Edit relations» affordance is rendered (additive wiring — zero read-view regression)
```

## Verification

## Notes

- **Premise correction (verify-before-assert):** the originating puzzle node stated «T3.1 added a discoverability ⋯-affordance in the read-view AssetRelationsTable». In fact T3.1 (#3764) built the Relations section *inside* `PropertyEditorModal` (opened via the `edit-properties` palette command); the read-view `AssetRelationsTable` had **no** edit affordance at all. This requirement fills exactly that read-view → editor entry-point gap.
- **Design (orchestrator-confirmed, block-level not per-row):** the affordance is block-level (edits the viewed asset's relations) — unambiguous and parity-safe. A per-row ✎ was rejected: a backlink row's relation is owned by a different asset, making per-row targeting architecturally ambiguous.
