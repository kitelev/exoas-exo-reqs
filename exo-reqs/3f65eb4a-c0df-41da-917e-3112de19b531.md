# req(exo): layout relation/query tables render task-link labels via the homoiconic `exo__DisplayNameSpec` system

> Source: follow-up of req `[[a577316f-f2ff-469b-a6d3-ff946eccc68f]]` (DailyNote tasks table → homoiconic displayName, PR #3862 → v16.178.0) + GitHub issue #3863. The DailyNote-table wiring deliberately did NOT thread the resolver through `UniversalLayoutRenderer`'s constructor (hot file). This req is that broader wiring: the layout relation/query tables (`RelationsRenderer` / `UniversalLayoutRenderer` / `TableLayoutRenderer`) still render task-link labels via plain `exo__Asset_label` with NO status/class prefix — so a Doing task shows no 🔄 there, unlike native links and (now) the DailyNote table. Route those cells through the same `DisplayNameResolver` stack.
>
> **Sibling of a577316f — identical mechanism, different surfaces.** The homoiconic vault specs already exist (12 `exo__DisplayNameSpec` on exoas-exo); this is pure wiring + a binding test.

## Job Story

When I open any asset whose layout shows a relations / children / query table with task links, I want each task's status prefix (🔄 while Doing, …) to be the same as on native links and in the DailyNote table — driven by my vault `exo__DisplayNameSpec` data — so there is no surface where Exocortex renders a task link without the prefix my spec declares.

## Statement (Gherkin)

```gherkin
Background:
  Given the vault exo__DisplayNameSpec "🔄 while Doing" for ems__Task (asset d069c316)
    And its ems__Effort_status matchPath / EffortStatusDoing matchValue conditional matcher (req ed4201d1)

Scenario: a layout relation/query table shows the homoiconic 🔄 for a Doing task
  Given an asset whose layout renders a relation / children / query table containing links to ems__Task assets
    And one linked task's ems__Effort_status is Doing (stored UID-canon [[<doing-uid>]] / [[<doing-uid>|label]])
  When the plugin renders that table
  Then that task link's cell shows "🔄 <label>" — resolved through DisplayNameResolver (the vault spec)
    And a non-Doing task link shows "<label>" (the conditional spec does not participate)
    And this matches what a native [[task]] link and the DailyNote tasks table show for the same task

Scenario: reuse the DailyTasksRenderer resolver helper — null-safe fallback, no regression
  Given the resolver is built from the plugin's printNameRuleService + displayNameSettings
  When printNameRuleService is absent (e.g. a unit-test mock) or the resolver yields nothing
  Then the cell falls back to the plain exo__Asset_label / basename (no crash, byte-identical to today)

Scenario: non-task links are unchanged unless a spec applies
  Given a relation table row linking to a non-Task asset (Area / Project / Concept) with NO applicable exo__DisplayNameSpec
  When the table renders
  Then that link's label is unchanged from today (its own class spec, or the plain label)
```

## Verification

**Revert-verified (component, production-shape):** the bound suite (`@req:3f65eb4a-c0df-41da-917e-3112de19b531`) drives the real relation/layout-table label path over an in-memory vault carrying the real 🔄-Doing `exo__DisplayNameSpec` + its parts, resolved via the real `DisplayNameResolver` + `PrintNameRuleService.scanVault` — no hand-injected label, no stubbed resolver.

Revert-verify axes (record in the PR body at implementation time):
