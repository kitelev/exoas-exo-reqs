# req(exo): wikilink display-label precedence (alias > exo__Asset_label > filename)

## Job Story

When I look at a wikilink, I want its visible label to follow a predictable
precedence — my explicit alias if I wrote one, otherwise the target's
`exo__Asset_label`, otherwise the bare filename — so that aliasing is purely a
display choice and never changes anything but the rendered text.

## Statement (Gherkin)

```gherkin
Scenario: explicit alias wins
  Given a wikilink "[[<uid>|<alias>]]" whose target has an exo__Asset_label
  When the live-preview plugin renders the link
  Then the plugin leaves it untouched and Obsidian shows "<alias>"

Scenario: no alias → target label
  Given a wikilink "[[<uid>]]" whose target has exo__Asset_label "<label>"
  When the plugin renders the link
  Then it displays "<label>"

Scenario: no alias and no/blank label → filename
  Given a wikilink "[[<uid>]]" whose target has no (or a blank) exo__Asset_label
  When the plugin renders the link
  Then it shows the bare filename ("<uid>")
```

## Notes

- **Already-satisfied (0 production code).** The precedence is plugin-owned and
  pre-exists; this requirement only binds it to the SDD ledger via a focused
  test that exercises the full alias > label > filename chain.
- **Plugin-owned boundary.** Obsidian renders the raw target; the plugin decides
  whether to relabel. The alias-wins step is the plugin EXCLUDING aliased links
  from relabel; the label/filename steps are `resolveLabel`.
- **Companion to the exocmd-resolution requirement** (9e84c833): together they
  state the full intent — an alias affects ONLY display, never exocmd
  resolution.
