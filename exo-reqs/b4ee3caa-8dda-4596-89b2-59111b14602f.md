# req(exo): displayName rendered from vault-declared exo__DisplayNameSpec (v1 thin, single-hop)

## Job Story

When I open or link an asset of any class, I want its displayName to be assembled from a
**vault-declared, queryable specification** of which properties/literals compose the name and in
what order — so that I (or a future maintainer) can change how a class prints its name by editing
RDF assets, not plugin TypeScript. In particular, a composite class like `ems__Statement` should
print `subject predicate object` from a structured spec, and the existing per-class suffixes
(TaskPrototype/MeetingPrototype/RFC) should keep rendering exactly as before, now driven by vault
data instead of hardcoded constants.

## Acceptance (Gherkin)

```gherkin
Feature: displayName is composed from a vault-declared exo__DisplayNameSpec (v1 thin, single-hop)

  Background:
    Given the plugin's DisplayNameResolver reads exo__DisplayNameSpec specifications from the vault
    And a spec is selected for an asset's class (exact class or engine class-hierarchy walk)
      choosing the highest exo__DisplayNameSpec_priority
    And a spec's exo__DisplayNamePart children are concatenated in exo__DisplayNamePart_order
    And an exo__PrintedProperty resolves its property single-hop (target's exo__Asset_label, no re-render)
    And an exo__PrintedLiteral emits its literal verbatim, and empty parts collapse

  Scenario: Suffix-class default renders from a vault spec (byte-identical to the old hardcode)
    Given a class TaskPrototype with a vault exo__DisplayNameSpec whose ordered parts are
      [ PrintedProperty(exo__Asset_label), PrintedLiteral(" (Task Prototype)") ]
    And no TS classTemplate exists for TaskPrototype
    When an asset of class TaskPrototype with label "Morning Routine" is rendered
    Then its displayName is "Morning Routine (Task Prototype)"

  Scenario: Composite class renders "subject predicate object" from structured parts
    Given a class ems__Statement with a vault exo__DisplayNameSpec whose ordered parts are
      [ PrintedProperty(subject), PrintedLiteral(" "), PrintedProperty(predicate),
        PrintedLiteral(" "), PrintedProperty(object) ]
    And the referenced subject/predicate/object targets carry ready exo__Asset_label values
    When an ems__Statement asset is rendered
    Then its displayName is "<subject-label> <predicate-label> <object-label>" (single-hop)

  Scenario: Higher-priority spec wins when two specs apply to the same class
    Given two exo__DisplayNameSpec assets both with exo__DisplayNameSpec_appliesToClass = ems__Task
    And their exo__DisplayNameSpec_priority values differ
    When an ems__Task asset is rendered
    Then the spec with the higher priority is applied

  Scenario: No spec → fallback chain
    Given an asset of a class that has no applicable exo__DisplayNameSpec
    When it is rendered
    Then its displayName is its exo__Asset_label, or its basename, or its UID (in that order)

  Scenario: Pure-label classes keep rendering their label after classTemplate removal
    Given the TS classTemplates for Task/Project/Area/Meeting have been removed
    And no vault exo__DisplayNameSpec applies to them
    When an ems__Task / ems__Project / ems__Area / ems__Meeting asset is rendered
    Then its displayName is its exo__Asset_label (unchanged from before)

  Scenario: Regression guard — engine must honour the vault spec (revert-verify RED direction)
    Given a class whose suffix behaviour is defined ONLY by a vault exo__DisplayNameSpec
      (its TS classTemplate has been removed)
    When the engine's spec-reading path is reverted so it ignores exo__DisplayNameSpec
    Then that class renders its raw exo__Asset_label instead of the composed suffixed name
    # This is the pre-fix behaviour the spec-reading removes (the revert-verify RED anchor).
```

## Notes

- **v1 = single-hop, non-recursive.** Verified on origin/main: `DisplayNameTemplateEngine.ts` (319 LOC)
  is single-pass — `render → resolveValue → formatWikilinkValue` reads the target's `exo__Asset_label`
  once and does NOT re-render the target's own template. Terminals in a Statement (Person/etc.) carry
  ready labels, so single-hop suffices. Recursion + cycle-guard + A↔B termination test = v2.
- **Anti-self-brick:** the engine must read `exo__DisplayNameSpec` BEFORE the TS `classTemplates` are
  removed, else assets briefly lose their names.
- **Delivery:** floor suffix defaults (TaskPrototype/MeetingPrototype) + migrated inbox__RFC rule ship
  as vault assets in `kitelev/exoas-exo` (bootstrap/pull), not bundled in the binary. Cold-start may
  need a min synchronous TS seed for the 2 floor suffix classes to avoid raw→suffixed flicker.
- **Platform exception (stays TS):** DailyNote `{{_basename}}` + the `_basename`/`_created` synthetic
  tokens (obsidian-calendar-plugin needs basename; created is engine context).
- **Mirrors** the homoiconic patterns of `exocmd__Command` / `exo__Layout` (presentation declared in
  the vault, engine interprets). Source onto-RFC `92b91345`; PMBOK project `47d57acf`.
