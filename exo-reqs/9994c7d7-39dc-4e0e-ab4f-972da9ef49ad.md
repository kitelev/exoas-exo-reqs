# req(exo): CLI vault adapter never re-nests the vault root (no phantom duplicate tree)

## Job Story

When a co-located asset create resolves a target folder that — through an
upstream folder resolver — comes out as the vault's own absolute path with its
leading slash stripped (`Users/kitelev/vaults/<vault>/assetspaces/...`), I want
the vault adapter to write the asset into its correct co-located folder and
create **no** phantom `<vault-root>/Users/...` duplicate directory tree — so the
vault root is never polluted with self-replicating empty mirror trees that can
confuse vault scanning and co-location resolution (issue #3761).

## Statement (Gherkin)

```gherkin
Feature: CLI vault adapter never re-nests the vault root

  Background:
    Given a vault rooted at an absolute path ROOT (e.g. "/Users/kitelev/vaults/vault-tbank")
    And a co-located ontology folder "assetspaces/kitelev/exoas-tbank/tbank-efforts" under ROOT

  Scenario: resolved target folder is the vault's absolute path with the leading slash stripped
    Given a folder resolver returned the co-located folder as ROOT's own absolute
          path with the leading slash stripped
          (i.e. "Users/kitelev/vaults/vault-tbank/assetspaces/kitelev/exoas-tbank/tbank-efforts")
    When the vault adapter writes the new asset "<uid>.md" into that folder
    Then the asset file lands at "ROOT/assetspaces/kitelev/exoas-tbank/tbank-efforts/<uid>.md"
    And NO directory tree is created under "ROOT/Users/..." (no phantom duplicate of the vault root)

  Scenario: ordinary vault-relative target folder is unaffected (regression guard)
    Given a folder resolver returned a normal vault-relative folder
          "assetspaces/kitelev/exoas-tbank/tbank-efforts"
    When the vault adapter writes the new asset "<uid>.md" into that folder
    Then the asset file lands at "ROOT/assetspaces/kitelev/exoas-tbank/tbank-efforts/<uid>.md"

  Scenario: a true-absolute target folder inside the vault is unaffected (regression guard)
    Given a folder resolver returned the absolute co-located folder
          "ROOT/assetspaces/kitelev/exoas-tbank/tbank-efforts"
    When the vault adapter writes the new asset "<uid>.md" into that folder
    Then the asset file lands at "ROOT/assetspaces/kitelev/exoas-tbank/tbank-efforts/<uid>.md"
```

## Design choice (chosen approach, self-approved)

Two layers could be guarded:

1. **Source** — stop `parentFolderOf` (core, shared by CLI + plugin) from
   stripping the leading slash. Rejected as the sole fix: it is shared with the
   Obsidian plugin, whose `vault.create` requires vault-**relative** paths, so
   "keep absolute" would be correct for the CLI's filesystem adapters but wrong
   for the plugin; and it would not protect future/edge upstreams.

2. **Chokepoint (chosen)** — harden the CLI write adapters' `resolvePath` to
   detect when a "relative" argument is in fact the vault's own absolute path
   with a stripped leading slash, and resolve it to the correct in-vault
   location instead of `path.join`-nesting it under the root. This is the single
   point ALL co-located CLI writes pass through, is robust to any upstream that
   produces such a path, matches issue #3761's explicit optional acceptance
   criterion ("a guard that detects and refuses to mkdir a target folder that
   re-nests the vault root path"), and does not touch the plugin.

The plugin-side surface (Obsidian provider returning an absolute path) is a
separate follow-up outside this CLI-scoped fix.

## Acceptance / verification

- Production-shape integration test driving the REAL `NodeFsAdapter` and
  `FileSystemVaultAdapter` against a real temp vault with the production-observed
  leading-slash-stripped absolute folder. Revert-verified: RED (doubled tree
  created) against the unguarded adapter, GREEN (correct co-located write, no
  doubled tree) with the re-nest guard.
