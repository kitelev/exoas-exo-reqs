# req(exo): `exo__DisplayNameSpec` conditional specialization (matchPath/matchValue, per-render matcher)

> Source: onto-RFC `92b91345` §«v2 conditional-срез — scope + worked example» (activated 2026-07-09 by first real-demand — Andrey: «🔄-префикс в displayName у задач в статусе Doing»). PMBOK project `fec2f499`. Conditional-slice only: single-hop matchPath + single matchValue + per-render matcher; multi-hop path / value-format / recursion / conjunctions / authoring-helper remain v2-deferred.

## Job Story

When I want a class's displayName to specialize on the instance's *state* (e.g. a 🔄 prefix on an `ems__Task` while it is `Doing`), I want to declare that condition **as vault data** on an `exo__DisplayNameSpec` (a `matchPath` property + a `matchValue`) — so the specialized name appears and disappears **automatically** as the instance's value changes, without any code change and without a separate template per state.

## Statement (Gherkin)

```gherkin
Scenario: a conditional spec applies only when the instance value matches
  Given an exo__DisplayNameSpec for class ems__Task with a HIGHER priority
    And its exo__DisplayNameSpec_matchPath points at the property ems__Effort_status
    And its exo__DisplayNameSpec_matchValue is the EffortStatusDoing enum asset
    And its parts are a PrintedLiteral "🔄 " (order 0) + a PrintedProperty exo__Asset_label (order 1)
  And an unconditional ems__Task spec (or the fallback) yields just the label
  When the plugin resolves the displayName of an ems__Task whose ems__Effort_status is Doing
  Then the displayName is "🔄 <label>" (the conditional spec wins by priority)
  When the plugin resolves the displayName of an ems__Task whose ems__Effort_status is NOT Doing
  Then the displayName is "<label>" (the conditional spec does not participate)

Scenario: the condition is evaluated per-render, not baked at compile time
  Given the conditional 🔄-Doing spec is loaded once via scanVault
  When the same task's ems__Effort_status changes from Doing to Done
  Then a re-resolve yields "<label>" (the 🔄 disappears) — no re-scan required
  When it changes back to Doing
  Then a re-resolve yields "🔄 <label>" again

Scenario: dual-IRI equality — the matcher matches UID- and label-form status values
  Given the conditional 🔄-Doing spec whose matchValue is the EffortStatusDoing UID
  When the task's ems__Effort_status frontmatter is the UID form [[<doing-uid>]]
  Then the matcher matches (🔄 shown)
  When the task's ems__Effort_status frontmatter carries the alias form [[<doing-uid>|EffortStatusDoing]]
  Then the matcher still matches (cleaned-identity equality, not raw-string)

Scenario: no matcher → v1 unconditional path is byte-identical (no regression)
  Given a spec with NO matchPath/matchValue (pure v1)
  When the plugin resolves any asset of that class
  Then selection is identical to v1 (highest-priority unconditional spec wins)
    And pure-label classes (Task/Project/Area/Meeting) and suffix classes
    (TaskPrototype/MeetingPrototype) render exactly as before
```
