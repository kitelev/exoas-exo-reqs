# req(exo): the Relations renderer fetches reified relations via a ctor-injected triple store, gated on readiness and filtered to relation predicates

## Job Story

When I open an asset that participates in reified `exo__Statement` relations, I want
the Relations renderer to be able to source those relations from the live triple
store — but only once the store has finished indexing (so a cold-start render never
shows a partial, under-counted set) and only for genuine relation predicates (not
`rdf:type` / system / plumbing noise) — so that the subsequent integration phase can
merge them into the unified relation list correctly and the marker phase can attribute
them, without the renderer ever leaking a half-indexed or noisy reified view.

## Statement (Gherkin)

```gherkin
Given a RelationsRenderer constructed with a live triple store, an isStoreReady probe,
  and the indexer's notePathToIRI (all new OPTIONAL ctor params; ExocortexPlugin.ts untouched)
  And a triple store holding reified exo__Statement instances for an asset A
When I call renderer.getReifiedRelationsGated(A)
Then while isStoreReady() returns false I get [] (the cold-start gate — never a partial
     under-counted set; the post-refresh re-render self-heals once indexing settles)
  And once isStoreReady() returns true I get A's reified relations from the statement instances
  And any reified statement whose _predicate is a non-relation predicate (rdf:type, an
     exo:Asset_*/Instance_/Class_/Statement_ system or plumbing predicate, or RDFS/OWL noise)
     is filtered out (predicate-whitelist by exclusion)
  And when the triple store or notePathToIRI dependency is absent I get [] (back-compat —
     existing callers behave exactly as before)
```
