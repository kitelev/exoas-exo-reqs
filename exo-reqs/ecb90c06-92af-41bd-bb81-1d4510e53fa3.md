# req(exo): `PreconditionEvaluator` `$today` (and its sibling date tokens) resolve to today's LOCAL calendar day (not UTC)

## Job Story

When a dynamic command's visibility/availability precondition compares an asset's date against `$today` in a precondition-SPARQL ASK, just after local midnight (my timezone is UTC+5, so UTC is still the previous calendar day), I want `$today` to resolve to **today's LOCAL date** — the same local basis as its sibling tokens (`$yesterday`, `$thisWeekStart`, …) and what a human reading their calendar sees — so a button's visibility never mis-fires at the day boundary and `$today` never absurdly equals `$yesterday`.

## Statement (Gherkin)

```gherkin
Scenario: PreconditionEvaluator $today just after local midnight in a UTC+N timezone returns the LOCAL day
  Given the process timezone is Asia/Almaty (UTC+5, no DST)
  And the wall clock is 2026-07-02T19:27:00Z (= 2026-07-03T00:27 local, local day 03, UTC day 02)
  When PreconditionEvaluator.substituteVariables interpolates "$today" into a precondition SPARQL
  Then the interpolated literal is "2026-07-03"^^xsd:date (today's LOCAL calendar day)
  And it is NOT "2026-07-02"^^xsd:date (the UTC day, which the former UTC form produced)
  And "$today" no longer equals "$yesterday" (which is "2026-07-02"^^xsd:date)

Scenario: a precondition ASK comparing an asset date against $today matches TODAY's LOCAL day at the boundary
  Given the same instant
  And an asset whose planned date triple is today's LOCAL day "2026-07-03"
  When PreconditionEvaluator.evaluate runs a precondition ASK that compares the asset date against "$today"
  Then the ASK matches (the command is available)
  And with the former UTC "$today" (= "2026-07-02") the ASK would NOT match
```

## Verification (plan)

**Binding class:** `Unit` — the fix is a leaf change confined to `PreconditionEvaluator.substituteVariables`. Both the string-substitution boundary case and a production-shape `evaluate()` ASK are exercised through the real `PreconditionEvaluator` (no mocks of the date basis). Test titles carry `@req:ecb90c06-92af-41bd-bb81-1d4510e53fa3`.

**Revert-verify** ([[integration-test-revert-verify]]): with `$today` reverted to `clock.now().toISOString().slice(0,10)` (and the siblings back on `ALMATY_OFFSET_MS`), the boundary test resolves `$today` to `"2026-07-02"` (== `$yesterday`) → **RED** (asserts `"2026-07-03"` and `$today !== $yesterday`); restored to the shared-local form → **GREEN**. CI-robustness ([[jest-timezone-sensitive-tests]]): `process.env.TZ` cannot be re-tzset at runtime under jest (V8 caches the worker timezone), and the fix has NO observable effect in a UTC runner — so the tests install a `Date` subclass (shared `installFakeOffsetDate` helper) simulating a fixed UTC+5 (Asia/Almaty, no DST) offset at `2026-07-02T19:27:00Z` (= 00:27 local, local day 03 / UTC day 02), independent of the runner's real timezone. A guard (`getHours()===0`, `getUTCDate()===2`) asserts the simulated tz is active so the tests can never silently pass both ways in a UTC-tz CI runner.
