# req(exo): bootstrap installs only the SDK floor — no --exocmd option

## Job Story

When I bootstrap a fresh vault, I want `exocortex bootstrap` to install **only the SDK platform** (`exo`, the TS-floor), so that the command stays conceptually clean — exocmd is an ordinary optional AssetSpace, added later via `exocortex assetspace add` (or by applying a profile), never bundled into the platform-install step.

## Rationale

`exocmd` is an **AssetSpace** (a "jar" in the SDK metaphor), not part of the **SDK/platform** (`exo`). The TS-floor is `{exo}` (code: `SDK_FLOOR` / `PLUGIN_UI_FLOOR` = `[{exo}]`, RFC `5aa2a73a` / #3440). A bootstrap command that offers a first-class `--exocmd` flag implies exocmd belongs to the platform-install step; removing the flag aligns the CLI with the floor model (decision: Andrey interview 2026-06-27).

## Statement (Gherkin)

```gherkin
Given the `exocortex bootstrap` CLI command
When I inspect its registered options
Then there is NO `--exocmd` option
  And the only required AssetSpace option is `--exo` (the SDK floor)
  And passing an unknown `--exocmd <url>` is rejected as an unknown option
```

## Verification

**Binding (unit):** `@req:e0e5ad0f-56ce-4bc8-9e1b-7a0e20a735d7` — a unit test asserting `bootstrapCommand()` registers no `--exocmd` option (`options.find(o => o.long === "--exocmd")` is `undefined`).

**Revert-verify (unit):** re-adding the `--exocmd` `.option(...)` makes `bootstrap.command.test.ts "registers NO --exocmd option"` go **RED** (option found); removing it → **GREEN** (2026-06-27, worktree origin/main `5032e8da`). Implemented in **PR #3745** (merge `dc83aed`, CI 14/14 green).

> Priority P2: a CLI conceptual-hygiene change (no Alpha-critical user-facing runtime behavior). A unit binding on the command's option set is sufficient below the P0 binding-class floor.
