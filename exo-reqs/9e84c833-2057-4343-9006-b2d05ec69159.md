# req(exo): alias in exo__Instance_class is display-only for exocmd resolution

## Job Story

When an `exo__Instance_class` wikilink carries a display alias
(`"[[<uid>|<alias>]]"`) — e.g. on a Meeting asset whose alias is the human
meeting title — I want the exocmd command buttons (and palette/CLI behaviour)
to be EXACTLY the same as if the link had no alias, so that an alias I added
purely for readability never silently removes my command buttons.

## Statement (Gherkin)

```gherkin
Scenario: an aliased instance-class resolves the same exocmd commands as alias-free
  Given an asset whose exo__Instance_class is "[[<uid>|<arbitrary display alias>]]"
    And a class-targeted CommandBinding exists for that class
  When the plugin resolves exocmd commands for the asset
  Then the resolved command set is IDENTICAL to the alias-free "[[<uid>]]" form
   And the command buttons render (a non-empty set)

Scenario: the alias never participates in resolution (display-only)
  Given the same asset with any display alias (the class name, an arbitrary
        title, a symbolic-form ref, or an array entry)
  When commands are resolved
  Then resolution depends ONLY on the wikilink target (before the "|"), never
       on the alias text
```

## Notes

- **Single fix-point.** Only `DynamicCommandButtonGroupBuilder.extractAssetClasses`
  extracts `exo__Instance_class` to drive binding-resolution. The palette and
  CLI surfaces do not, so the fix is one closure change.
- **Empirical nuance.** The legacy `matchesReference` #2740 cross-match + the
  resolver's pipe-stripping `normalizeWikilink` already handle a UID-form alias
  that equals the class name and any symbolic-form alias; the real bug is a
  UID-form ref with an arbitrary (non-class) display alias. The fix makes the
  alias display-only for ALL forms, end-to-end.
- **No behavioural change for alias-free / symbolic forms** — they already
  resolved; the fix only repairs the arbitrary-UID-alias case while keeping the
  rest working (regression-covered).
