# req(exo): cli create co-locates a new asset into its isDefinedBy ontology folder (not the inbox)

## Job Story

When I create a new asset whose `exo__Asset_isDefinedBy` points at an ontology that exists on disk, I want the file written next to that ontology (the co-location invariant) rather than dumped in the inbox, so my vault stays organised and the asset is immediately discoverable under its ontology — and when isDefinedBy is missing or unresolvable, I want a safe inbox fallback rather than a vault-root write.

## Statement (Gherkin)

```gherkin
Given a vault with an ontology file whose UID is referenced by a new asset's exo__Asset_isDefinedBy
When I run `cli create` with --property exo__Asset_isDefinedBy=[[<ontology-uid>]]
Then the asset is written into that ontology's folder (not 01 Inbox)
And the file physically exists in the co-located folder and NOT in the inbox

Given a new asset whose exo__Asset_isDefinedBy is missing, `!`-prefixed, or points at an unresolvable UID
When I run `cli create`
Then the asset fails open to the 01 Inbox default folder
```
