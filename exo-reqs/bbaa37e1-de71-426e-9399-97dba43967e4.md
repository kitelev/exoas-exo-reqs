# req(exo): a property_set grounding with targetValueQuery computes the property value from a NamedQuery and writes it

## Job Story

When a command needs to set a property to a value that has to be *computed* from the graph (e.g. "the archive ontology for this asset"), I want a `property_set` grounding to run a referenced NamedQuery read-only — with the current asset auto-injected — and write the computed scalar (an IRI wrapped as a wikilink) into the target property, so command behavior stays homoiconic (the query lives in a vault asset) and CLI and UI share one read-side path.

## Statement (Gherkin)

```gherkin
Given a property_set grounding whose targetProperty is exo__Asset_isDefinedBy and whose targetValueQuery references a query__NamedQuery
And that NamedQuery's ```sparql block walks $currentAsset --isDefinedBy--> srcOntology --archiveOntology--> archiveOntology
And a target asset whose exo__Asset_isDefinedBy points at the source ontology
When I run `cli apply <command> <target>`
Then the executor runs the NamedQuery with $currentAsset = the target, takes the scalar IRI result, wraps it as a wikilink, and writes it
And the target's exo__Asset_isDefinedBy is re-anchored to "[[<archiveOntology-uid>]]"
And the source-ontology reference is gone from the file
```
