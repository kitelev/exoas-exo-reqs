# req(exo): validate schema does NOT emit a false sh:class violation for a metaclass-typed instance (Issue #3247)

## Job Story

When a class file is typed as exo__Class but never declares its own `exo__Class_superClass: [[exo__Asset]]` (common under UID-canon), I want schema validation to still treat instances of that class as conforming to an exo__Asset-ranged property, so that a missing-but-derivable superClass declaration does not flood the report with false violations and hide the real ones.

## Statement (Gherkin)

```gherkin
Given a class C typed as exo__Class with NO own exo__Class_superClass declaration
And exo__Class is itself subClassOf exo__Asset
And an asset instance typed as C
And a reference asset with exo__Asset_relates: [[instance]] (shape range = exo__Asset)
When I run `validate schema --shapes-mode` over the vault
Then the metaclass-propagation pass infers C subClassOf exo__Asset
And NO sh:class violation is emitted against exo__Asset for the instance on the exo__Asset_relates path
```
