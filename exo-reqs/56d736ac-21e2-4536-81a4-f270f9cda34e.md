# req(exo): cli apply emits exo__Instance_class as UID-form wikilink (parity with the UI button) when the class TBox is present

## Job Story

When I run a create_instance command through the CLI (e.g. for automation or parity testing), I want the new instance's `exo__Instance_class` written as the canonical UID-form wikilink — exactly what the in-app button produces — so the asset's class edge resolves identically across both runtimes and UID-canon is never silently broken by a label-form reference.

## Statement (Gherkin)

```gherkin
Given a vault containing a create_instance command whose grounding declares a label-form targetClass (e.g. "ems__Task")
And the class TBox file (carrying the canonical UID and label "ems__Task") is present in the vault
When I run `cli apply <command> <target>` to create the instance
Then the created instance's exo__Instance_class is written in UID-form ("[[1b20a8f0-...]]")
And the frontmatter contains no label-form class wikilink ("[[ems__Task]]")

Given the class TBox file is absent from the vault
When I run `cli apply` for the same command
Then the resolver returns null and the original label-form ("[[ems__Task]]") is preserved (backward-compatible)
```
