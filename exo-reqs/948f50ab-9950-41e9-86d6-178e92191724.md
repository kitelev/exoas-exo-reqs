# req(exo): generic Settings Distribution engine — declared-key export/import round-trip, allowlist-by-construction

## Job Story

When I snapshot my plugin settings to vault assets (Export) and later restore them (Import) — on desktop or iPhone — I want exactly my DECLARED settings round-tripped (export → restore yields the same live value), and I want the engine to be physically incapable of leaking an undeclared key (`pat`, `logChannels`, …) into the shared assets — so settings distribution is a single homoiconic mechanism with a correctness guarantee (allowlist-by-construction), not a per-source bespoke writer.

## Statement (Gherkin)

```gherkin
Given a SettingsSource that declares N keys and holds live values for them
When I exportSettings(source)
Then exactly N setting__Setting asset specs are produced — one per declared key — and nothing else
And each carries the declared key (setting__Setting_key) and the datatype-rendered live value (setting__Setting_value)

Given those exported assets, and the live values have since been changed
When I importSettings(source, assets)
Then every declared key's live value is restored to the exported snapshot

Given an asset whose setting__Setting_key is NOT in the source's declared set
When importSettings runs
Then that asset is SKIPPED (its value is never written to live settings) — allowlist-by-construction

Given the live keyspace also carries an undeclared field (e.g. a secret)
When exportSettings runs
Then that field is never read and never appears in any exported asset — allowlist-by-construction
```
