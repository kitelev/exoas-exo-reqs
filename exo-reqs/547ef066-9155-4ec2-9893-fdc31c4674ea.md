# req(exo): the requirements-audit checker recognizes `component` as a distinct binding-class below the P0 real-prod floor

## Job Story

When I tag a requirement's test binding as a **component** test (`.test.tsx` /
jsdom — a UI component exercised in isolation under a synthetic DOM, not live
Obsidian), I want the traceability checker to recognize `component` as its own
binding-class tier — distinct from `unit`, but still **below** the real-prod
floor — so that component coverage is reported honestly and a P0 requirement
cannot be "satisfied" by a jsdom-only binding.

## Statement (Gherkin)

```gherkin
Scenario: component is a distinct binding class, classified below the P0 real-prod floor
  Given a requirement declares bindingClass "[[...|req__RequirementBindingClassComponent]]"
  When `exocortex requirements audit` parses its binding classes
  Then the parsed binding class is "component" (a distinct token, NOT collapsed into "unit")
  And a P0 requirement bound SOLELY to "component" is reported as a binding-class floor violation
  And a P0 requirement bound to ["component"] together with "unit" is STILL a floor violation
  And a P0 requirement bound to ["component", "integration"] is NOT a floor violation
    (the real-prod integration binding satisfies the floor)
```

## Verification

**Binding (unit-class):** `@req:547ef066-9155-4ec2-9893-fdc31c4674ea` on the unit
test of the pure `parseBindingClasses` + `auditTraceability` floor logic in
`packages/cli/tests/unit/commands/requirements-audit.test.ts`.

**Revert-verified** ([[integration-test-revert-verify]]): the cited test
exercises the genuine below-floor classification of `component`. Reverting the
production classification — temporarily adding `"component"` to
`REAL_PROD_CLASSES` — makes "a P0 requirement bound solely to component → floor
violation" FAIL (no violation reported); restoring (component absent from
REAL_PROD_CLASSES) → PASS. (Evidence recorded in the implementing PR body.)
