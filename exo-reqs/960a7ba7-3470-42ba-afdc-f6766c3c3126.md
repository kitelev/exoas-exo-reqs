# req(exo): CLI `resolve-buttons` — the actual command button-set oracle

> Source: GitHub issue `#3833` (extracted from superseded onto-RFC `03b0241d`; the CLI-verification axis Andrey chose to carry forward, 2026-07-05). Design direction (#5 — a CLI command) pre-approved by Andrey.

## Job Story

When I curate the command-button layout of a class (e.g. hiding planning commands off `ems__TaskPrototype` via a not-a-prototype precondition), I want a CLI command that prints the exact button-set the plugin would render on a given asset — so a layout audit is a single deterministic command instead of "open the live layout, eyeball it, and `apply --dry-run` each command", and so a curated layout is regression-testable.

## Statement (Gherkin)

```gherkin
Scenario: the button-set is binding-match INTERSECTED with precondition-eval
  Given an asset whose class hierarchy matches some exocmd__CommandBinding targetClass
  When I run `resolve-buttons <target> --vault <v>`
  Then the visible set is exactly the commands whose binding matches (Layer A,
    via CommandResolver.resolveForAssetMulti over the class hierarchy)
    AND whose precondition evaluates true (Layer B, via PreconditionEvaluator.evaluate)
  And a command bound only to an UNRELATED class never appears (Layer A excludes it)
  And this mirrors the plugin's DynamicCommandButtonGroupBuilder.resolveViaFullPath path

Scenario: class-curation precondition hides a bound command (prototype layout)
  Given a prototype-class asset that inherits its base class's bound commands
  And some of those commands carry a "not-a-prototype" precondition
  When I run `resolve-buttons` on the prototype
  Then the not-a-prototype-gated commands are HIDDEN (precondition false)
  And the unconditional commands remain visible
  And the result matches the live curated layout (not the pre-curation set)

Scenario: --show-hidden gives the binding-vs-precondition diagnostic dry-run cannot
  Given a command that binds to the target's class but whose precondition is false
  When I run `resolve-buttons <target> --show-hidden`
  Then that command is listed under "hidden" with reason "precondition-false"
  And this distinguishes a bound-but-hidden command from a never-bound one —
    a distinction `apply --dry-run` (precondition-only) cannot make

Scenario: deterministic ordering for snapshot-testable layouts
  Given a target with several bound-and-visible commands
  When I run `resolve-buttons` twice
  Then the visible order is identical and follows the resolver's
    (priority, depth, order) sort — so a layout can be snapshot-asserted
```
