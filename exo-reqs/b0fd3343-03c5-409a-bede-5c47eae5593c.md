# req(exo): ExoSync SHACL merge-gate blocks a merged asset that violates an intrinsic constraint (never ships an invalid auto-merge)

## Job Story

When ExoSync auto-merges two divergent edits of the same asset, I want the merged result to be checked against the schema's intrinsic constraints (required properties, cardinality, datatype) BEFORE it ships — and refused (routed to quarantine) if it would be invalid — so an automatic merge can never silently produce and propagate a schema-broken asset.

## Statement (Gherkin)

```gherkin
Given a merged candidate asset typed as ems__Task
  And the SHACL shapes require ems__Effort_status with minCount 1 (sh:Violation)
  And the merged candidate is missing ems__Effort_status
When the ExoSync merge-gate checks the merged asset
Then the gate verdict is ok:false with a minCount violation
And the invalid merge is refused (routed to quarantine), never shipped
```

## Verification

**Revert-verified (unit binding):** `@req:b0fd3343-03c5-409a-bede-5c47eae5593c` — disabling the intrinsic-violation collection in `MergeShaclGate.checkMergedAsset` (skipping `violations.push(v)` for non-class `sh:Violation`s, leaving the class-range branch intact) makes `MergeShaclGate.test.ts "minCount violation on the merged asset fails the gate"` go **RED** — the gate returns `ok:true` (an invalid merge would ship) instead of `ok:false`; restored → **GREEN** with the full suite (10/10, the open-world class-range tests stay GREEN — clean isolation) (2026-06-21, al-reqmgmt-p2-inc3, origin/main `53519a89`).

**Integration binding (revert-verified, CI-reliable):** `packages/core/tests/integration/sync/syncEngine.shacl-merge-gate.integration.test.ts` composes the **real** `GatedStructuredMerger` + **real** `MergeShaclGate` inside a full `SyncEngine` cycle against the production-shape in-memory `FakeGitHubRepo` (mock-github transport — **NO network**) and proves the D17 invariant end-to-end: a structurally-clean auto-merge whose merged result drops a minCount-required property (`ems__Effort_status`) is **refused → quarantined, never shipped to the remote**; a content-control merge that keeps the property ships normally (the verdict is content-dependent, not an always-quarantine stub). Revert-verify (2026-06-22, al-reqmgmt-floor, origin/main `f9a7ae81`): the gate-always-allow break of `MergeShaclGate.checkMergedAsset` makes the block scenario **RED** (the invalid merge ships) while the valid-merge control stays **GREEN**; restored → **GREEN** (2/2). This supersedes the earlier-cited `syncEngine.real-github.integration.test.ts`, which is network/PAT-gated **and** does not inject a gated merge layer at all (its `SyncEngine` runs the default gate-less `EMPTY_MERGE` path, so it never exercised this gate).

> **⛔ Status held at `approved` — NOT advanced to `active` (al-reqmgmt-floor, 2026-06-22).** Verify-before-assert finding: the SHACL merge-gate is **NOT wired into any production sync path** — both the plugin (`SyncDepsFactory`) and the CLI (`exosync-sync`) compose `GatedStructuredMerger` **without** a gate ("SHACL merge-gate deferred" follow-up), and there is **zero `new MergeShaclGate` call-site** in production `src`. The harness above composes the gate **explicitly** and revert-verifies its **blocking capability** in the full pipeline (so wiring it into the default engine is a drop-in), but the behaviour "ExoSync blocks an invalid auto-merge" is **not yet in force** in production — advancing to `active` now would be a **false-active**. The remaining step to `active` is wiring the real gate into the default `SyncEngine` composition (plugin-side `ShapeRegistry`/`ClassHierarchy`/`triplesFor`) — a separate, scoped behaviour-change with its own regression surface.
