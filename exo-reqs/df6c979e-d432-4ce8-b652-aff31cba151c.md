# req(exo): CLI `audit ontology-url` flags exocortex.my ontologies whose `exo__Ontology_url` lacks a trailing `#`

## Job Story

When I create a new ontology by copy-from-neighbour (or edit an existing one) and its
`exo__Ontology_url` accidentally drops the trailing `#` separator, I want a CLI
audit gate to catch it in CI — so the term-IRI concatenation (`url + localName`) and
SPARQL `PREFIX` expansion stay correct and the manual 2026-07-04 normalization of ~80
ontologies never has to be redone.

## Statement (Gherkin)

```gherkin
Given an ontology asset with an exo__Ontology_url in the exocortex.my namespace
      WITHOUT a trailing "#" (e.g. "https://exocortex.my/ontology/ems")
When `exocortex audit ontology-url --vault <path>` runs
Then a violation is flagged naming the file, the current URL and the expected
     canonical form ("https://exocortex.my/ontology/ems#")
  And the process exits with code 1

Given an ontology asset whose exo__Ontology_url is a FOREIGN vocabulary
      (host is not exocortex.my, e.g. "https://www.w3.org/2000/01/rdf-schema#")
When the audit runs
Then that ontology is SKIPPED (its author defines the form; "/" or "#" both valid)
  And it is counted under the "foreign-vocab" skip reason, never as a violation

Given a sub-ontology whose exo__Ontology_url has a hierarchical path
      ("https://exocortex.my/ontology/kitelev-period/quarters#")
When the audit runs
Then it PASSES (the path hierarchy is preserved; only the trailing "#" is required)

Given a vault where every exocortex.my exo__Ontology_url ends with "#"
When the audit runs
Then zero violations are reported and the process exits 0

Given a test ontology whose trailing "#" has been REMOVED
When the audit runs
Then it is flagged RED (violation);
  And when the "#" is restored the audit is GREEN (zero violations)
```
