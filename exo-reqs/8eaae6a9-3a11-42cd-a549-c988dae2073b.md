# req(exo): CLI `apply <cmd> --json` surfaces composite create-as-side-effect assets

## Job Story

When I drive `exocortex apply <composite-create-cmd> <target> --json --yes` (a command
whose grounding is a `composite` that creates a task/asset as a side effect), I want the
`created[]` array on stdout to report that created asset's identity (uuid + path + label)
â€” just like a direct `create-task-instance` already does â€” so a script can link/verify
what each apply produced without a separate SPARQL round-trip. Today a composite yields an
empty `created` array because `executeComposite` drops the created path.

## Statement (Gherkin)

```gherkin
Given a command whose grounding is a `composite` that includes a `create_instance` step
      (creates an asset as a side effect)
When `exocortex apply <cmd> <target> --vault <v> --json --yes` runs and succeeds
Then stdout is a single valid JSON object carrying `command`, `target` and a `created`
     array with one entry per asset the composite wrote to disk
  And JSON.parse(stdout).created[0] carries `uuid`, `path` and `label`
  And created[0].path resolves to a real file on disk
  And created[0].uuid equals that file's exo__Asset_uid (its UUID-canon basename)
  And created[0].label equals that file's exo__Asset_label (read fresh from disk)

Given a composite command that creates NO asset (e.g. a property_set-only composite)
When run with --json and it succeeds
Then stdout is a single valid JSON object whose `created` array is empty

Given a DIRECT create_instance command (e.g. create-task-instance) â€” #3906/req 55a79515
When run with --json
Then its behavior is unchanged: created[] still surfaces its single created asset

Given a composite command invoked in the plugin via a button (CommandExecutionFlow)
When it runs and creates an asset as a side effect
Then the plugin does NOT auto-open the created file in a new tab
     (composite `openPath` stays unset â€” plugin behavior unchanged)

Given the production surfacing is reverted so a composite never reports createdPaths
When the @req integration test runs
Then the composite created[0] assertion is RED; restoring the surfacing turns it GREEN
```

## Design (informative)

- Core `GroundingExecutor.ExecutionResult` gains a `readonly createdPaths?: readonly string[]`
  field â€” the vault-relative paths of assets created by a composite's create_instance
  step(s). Distinct from `openPath` (which the presentation layer opens in a tab):
  `createdPaths` is a surfacing-only channel; the composite leaves `openPath` UNSET so
  `CommandExecutionFlow` (plugin tab-open) is unchanged.
- `executeComposite` accumulates each step's `openPath` (and any nested-composite
  `createdPaths`) into a `createdPaths` array â€” purely additive; the existing
  `lastCreatedPath` threading to `body_template`/`targetsCreatedInstance` steps is
  byte-identical. Returns `{ success: true }` (no `createdPaths` key) when it created
  nothing â†’ today's output unchanged.
- CLI `apply.ts` maps `openPath` (direct) + each `createdPaths` entry (composite) into the
  `created[]` array, reading each created file fresh from disk for uuid + label (the same
  `getAbstractFileByPath`/`getFrontmatter` fresh-read as #3906).
