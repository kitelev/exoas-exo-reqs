# req(exo): parse-time `$tomorrow` resolves to the next LOCAL calendar day (not UTC)

## Job Story

When I press «Следующая итерация» on a `ems__WaitingCheckTask` just after local midnight (my timezone is UTC+5, so UTC is still the previous calendar day), I want the new iteration's `ems__Effort_plannedStartTimestamp` to be **tomorrow's LOCAL date** — the soft daily tickler must re-surface the check on the NEXT local day, not resurface it TODAY — so the «check again tomorrow» semantics (Vision Lock 5) hold regardless of the wall-clock hour or my timezone offset.

## Statement (Gherkin)

```gherkin
Scenario: $tomorrow just after local midnight in a UTC+N timezone advances to the next LOCAL day
  Given the process timezone is Asia/Almaty (UTC+5, no DST)
  And the wall clock is 2026-07-02T19:27:00Z (= 2026-07-03T00:27 local, local day 03, UTC day 02)
  When a $tomorrow SubstitutionToken PropertyDefault is resolved by the parse-time resolver
  Then the resolved value is "2026-07-04" (the next LOCAL calendar day)
  And it is NOT "2026-07-03" (the local TODAY, which the former UTC form produced)

Scenario: parity with the registry resolver
  Given the same instant
  When both the parse-time resolver (parseTimeResolve) and the registry resolver (makeDateResolver(1, "YYYY-MM-DD")) resolve $tomorrow
  Then both return the same next-local-day YYYY-MM-DD string
```

## Verification (plan)

**Binding class:** `Unit` — the fix is a leaf arithmetic change in `CommandResolver.parseTimeResolve`, exercised production-shape through the real loader `CommandResolver.loadCommand` (a real `InMemoryTripleStore` seeds the `$tomorrow` token + PropertyDefault + create_instance grounding, and the resolved PropertyDefault value is asserted). Test title carries `@req:bca93bfb-b663-4309-a19e-996de8e526da`.

**Revert-verify** ([[integration-test-revert-verify]]): with `parseTimeResolve` case `tomorrow` reverted to the UTC form (`setUTCDate` + `toISOString`), the test (TZ forced to Asia/Almaty, fake clock at `2026-07-02T19:27:00Z`) resolves to `"2026-07-03"` → **RED** (asserts `"2026-07-04"`); restored to the local form → **GREEN**. The test forces `process.env.TZ = "Asia/Almaty"` and guards that the timezone actually took effect (`getHours()===0`, `getUTCDate()===2`) so it can never silently pass in a UTC-tz CI runner.
