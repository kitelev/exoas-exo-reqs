# req(exo): composite step opt-in threading of the created instance to later mutating steps

## Context

The homoiconic command engine executes a `composite` grounding
(`exocmd__GroundingTypeComposite`) as an ordered list of `exocmd__Grounding_steps`.
`GroundingExecutor.executeComposite` tracks the most-recently-created asset's path
(`lastCreatedPath`, populated from a `create_instance` step's `openPath`) and threads
it to a later step — but **only** when that step's type is `body_template`. Every other
step type (`property_set`, `property_delete`, `property_append`, `property_increment`,
`property_shift`, `workflow_transition`, `service_call`) always receives the composite's
original click-target `filePath`.

## The gap (#3867)

You cannot compose `[create_instance, property_set]` so a **newly-created** asset is
mutated by the `property_set` — the mutator applies to the click-target (e.g. the
prototype the button was clicked from), which would wrongly mutate the prototype. The
concrete blocked use case: `[create-task-instance, move-to-backlog]` to reuse the
existing `move-to-backlog` `property_set` grounding instead of duplicating its
`ems__Effort_status = Backlog` literal into a bespoke `create_instance` `propertyDefault`.

## Design (opt-in, backward-safe)

A per-step boolean flag `exocmd__Grounding_targetsCreatedInstance: true` (domain field
`targetsCreatedInstance`, boolean-coerced by `CommandResolver.loadGroundingDefinition`
exactly like `cloneTargetBody`/`prefillLabelWithDate`). When a composite step carries
the flag AND a prior `create_instance` step has produced an asset, `executeComposite`
runs that step with `filePath = lastCreatedPath`. `targetIRI` stays the click-target
(so `$target` = source; the created asset is not yet indexed in the store). Flag
absent/false → click-target (default, zero regression).

## Acceptance criteria (BDD)

```gherkin
Feature: composite grounding — thread created-instance to later mutating steps

  Scenario: property_set step opts in to target the just-created asset
    Given a composite grounding with steps:
      | 1 | create_instance (creates a new asset, returns openPath) |
      | 2 | property_set with exocmd__Grounding_targetsCreatedInstance: true |
    When the composite executes against a click-target file
    Then step 2 writes its property into the CREATED asset's file
    And the click-target file is left unmodified by step 2

  Scenario: existing click-target-targeting composites keep working unchanged
    Given a composite grounding with steps:
      | 1 | create_instance (creates a new asset) |
      | 2 | property_set WITHOUT targetsCreatedInstance |
    When the composite executes against a click-target file
    Then step 2 writes its property into the CLICK-TARGET file (unchanged default)
    And this matches the current WaitingCheckTask "Следующая итерация" behavior

  Scenario: body_template threading is unaffected
    Given a composite grounding with steps: create_instance, body_template
    When the composite executes
    Then body_template still writes into the created asset's file (as today)
```

## Revert-verify

Through the real `GroundingExecutor.executeComposite` path (production-shape — real
`create_instance` step writes an asset, real `property_set` step mutates a file):
- **GREEN (with fix):** flag on step 2 → property in the CREATED file; click-target
  untouched by step 2.
- **RED (without fix):** neutralise the flag-consuming branch → the opt-in step writes
  into the CLICK-TARGET (created asset unmodified) while the default-behavior test stays
  GREEN.
