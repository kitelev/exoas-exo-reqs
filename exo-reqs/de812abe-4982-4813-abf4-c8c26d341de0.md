# req(exo): distribution-ontology meta-setting — homoiconic target, resolved from the graph

## Job Story

When the settings-distribution engine needs to know WHICH AssetSpace to export a
domain's settings into (or import them from), I want that target recorded as a
homoiconic meta-setting on the floor — so the distribution target is
user-configurable data in the graph, not a TS-hardcoded ontology UID — and I want
the engine to resolve it from the graph, falling back to picking by convention
when no meta-setting has been recorded yet.

## Statement (Gherkin)

```gherkin
Given a set of floor setting-assets that includes a setting__Setting whose
  setting__Setting_key references the distribution-ontology meta key (by UID)
  and whose setting__Setting_value is the AssetSpace UID "X"
When resolveDistributionOntology(assets, keyUid) runs
Then it returns "X"

Given a set of floor setting-assets with NO meta-setting for that key
When resolveDistributionOntology(assets, keyUid) runs
Then it returns null (the caller then picks the target by location-convention)

Given a meta-setting for that key whose value is blank / whitespace-only
When resolveDistributionOntology(assets, keyUid) runs
Then it returns null (a blank target is not a recorded choice)

Given a meta-setting authored with the legacy exo__Setting_key / exo__Setting_value predicates
When resolveDistributionOntology(assets, keyUid) runs
Then it still resolves the value (dual-read), so legacy-authored meta-settings keep working
```
