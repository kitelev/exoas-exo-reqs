# req(exo): configured `exocortex` code-block query tables render task-link labels via the homoiconic `exo__DisplayNameSpec` system

> Source: follow-up #3870 of req `[[3f65eb4a-c0df-41da-917e-3112de19b531]]` (#3863, relation/query tables → v16.179.0) — itself a sibling of `[[a577316f-f2ff-469b-a6d3-ff946eccc68f]]` (DailyNote table → v16.178.0). #3863 routed the relation/children/query tables fed by `RelationsRenderer.getAssetRelations`. This req closes the LAST surface: the configured `exocortex` **code-block** `TableLayout` tables (`TableLayoutRenderer` via `LayoutService` / `LayoutCodeBlockProcessor`) still resolve cell labels through the shared `wikilinkResolver.getAssetLabel(path)` property-value resolver (`LayoutCodeBlockProcessor.ts` ~line 408), NOT through `getAssetRelations` — so a Doing task link in a user-authored `exocortex` code-block table still shows the plain label with no 🔄.
>
> **Third and final sibling — identical mechanism, the general property-value wikilink cell path.** The homoiconic vault specs already exist (12 `exo__DisplayNameSpec`); this is wiring + a binding test. After this, ALL task-link render surfaces are homoiconic.

## Job Story

When I author an `exocortex` code-block query table on any note and it lists task links, I want each task's status prefix (🔄 while Doing, …) to match native links and every other Exocortex table — driven by my vault `exo__DisplayNameSpec` data — so there is NO remaining surface where a task link renders without the prefix my spec declares.
