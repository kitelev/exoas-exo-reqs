# req(exo): the Relations read-path helper sources reified relations from exo__Statement instances (outgoing + incoming, dual-IRI union, provenance)

## Job Story

When I open an asset A that participates in reified `exo__Statement` relations (A is
a statement's subject or object), I want the Relations read-path to surface those
relations from the statement **instances** — each with the resolved
subject/predicate/object and the backing statement asset's provenance — so that a
later phase can render reified relations as first-class, correctly attributed
edges (the precondition for the "reified · &lt;AS&gt;" marker and reify-editing),
instead of them being invisible or showing up as raw `exo__Statement_subject`
backlinks.

## Statement (Gherkin)

```gherkin
Given a triple store holding reified exo__Statement instances (each emitting the
  raw _subject/_predicate/_object triples AND the materialized D5 logical edge)
  And an asset A that is the subject of some statements and the object of others
When I call getReifiedRelations({ file: A, store, notePathToIRI })
Then I get one ReifiedRelation per statement instance referencing A
  And each carries direction (outgoing when A is _subject, incoming when A is _object),
      the resolved subject/predicate/object, objectIsLiteral, and the backing
      statement asset's path + AssetSpace (provenance from the statement, not the edge)
  And a statement whose _subject resolved to A's SYMBOLIC IRI (because A's label is a
      prefix__LocalName class reference) is still found via the dual-IRI union — not
      silently zero
  And the bare materialized edge is never surfaced as an extra relation
  And an asset with no statements (or an empty store) yields []
```
