# req(exo): CLI `apply <cmd> <target> --json` emits created uid/path

## Job Story

When I drive `exocortex apply create-task-instance <prototype> --json --yes` in a loop
to seed N task instances from a prototype, I want each call to report the created
asset's identity (uuid + path + label) directly on stdout — so I can link/verify what
each apply produced without a separate SPARQL query, since the grounding executor
already knew the created path.

## Statement (Gherkin)

```gherkin
Given a create_instance-grounding command (e.g. create-task-instance, grounding-type
      4367e2d6-6c92-450a-becb-abce1fb07682) bound to a prototype asset
When `exocortex apply <cmd> <prototype> --vault <v> --json --yes` runs and succeeds
Then stdout is a single valid JSON object carrying `command`, `target` and a `created`
     array whose entry includes `uuid`, `path` and `label` for the asset actually written
  And JSON.parse(stdout).created[0].path resolves to a real file on disk
  And JSON.parse(stdout).created[0].uuid equals that file's exo__Asset_uid (its UUID basename)
  And JSON.parse(stdout).created[0].label equals that file's exo__Asset_label

Given the same command run WITHOUT --json
When it succeeds
Then output stays backward-compatible: the existing human-readable success message,
     optionally with the created path appended
  And no JSON is written to stdout

Given a command whose grounding creates NO asset (e.g. a property_set command) run with --json
When it succeeds
Then stdout is a single valid JSON object whose `created` array is empty

Given --json mode
When the run completes
Then the human-readable success and multi-target summary notices are NOT written to
     stdout (so stdout parses cleanly as one JSON document); errors still go to stderr

Given the production behavior is reverted so apply never surfaces a created asset
When the @req integration test runs
Then the created[0] assertion is RED; restoring the surfacing turns it GREEN
```

## Envelope shape (informative)

```json
{
  "command": "create-task-instance",
  "target": "path/to/prototype.md",
  "created": [
    { "uuid": "...", "path": "path/to/new-task.md", "label": "..." }
  ]
}
```
